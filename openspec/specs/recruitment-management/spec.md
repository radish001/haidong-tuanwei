## MODIFIED Requirements

### Requirement: 招聘信息维护
系统 MUST 允许管理员以与青年信息库一致的列表布局和抽屉流程对招聘岗位信息执行新增、编辑、列表查询和删除操作，并要求岗位使用受控企业、经验、薪资、三级区域代码以及多选的专业要求、学历要求、学校类别、学校标签录入；招聘维护流程必须以更接近目标设计稿的筛选栏、表格布局和抽屉化表单视觉呈现。招聘新增和编辑表单中的“工作地区”控件 MUST 与其他常规表单字段使用一致的列宽规则，不得长期以明显更长的整行控件破坏表单对齐。招聘新增和编辑表单中的“专业要求”“学历要求” MUST 提供显式的“不限”选项，并将其视为当前“空要求表示不限”语义的可见表达。

#### Scenario: 新增招聘岗位
- **WHEN** 管理员在招聘列表页点击新增并提交合法的招聘岗位表单
- **THEN** 系统保存岗位信息
- **AND** 系统在招聘列表中展示该岗位

#### Scenario: 删除招聘岗位
- **WHEN** 管理员删除一条已存在的招聘岗位信息
- **THEN** 系统将该岗位从当前招聘列表中移除

#### Scenario: 招聘列表使用统一布局与分页交互
- **WHEN** 管理员进入招聘信息列表页
- **THEN** 系统必须在页面上方展示查询筛选区
- **AND** 系统必须在查询区域下方右侧展示新增、删除、匹配学生等操作区
- **AND** 系统必须提供页号切换、每页条数切换与局部刷新能力

#### Scenario: 招聘表单通过右侧抽屉承载
- **WHEN** 管理员在招聘列表页点击新增或编辑
- **THEN** 系统必须在当前页面右侧滑出抽屉面板承载岗位表单，并以更接近目标设计稿的视觉布局展示岗位表单、企业关联字段和操作按钮
- **AND** 系统不得通过整页跳转打开独立表单页

#### Scenario: 删除招聘前要求确认
- **WHEN** 管理员删除一条招聘岗位信息
- **THEN** 系统必须先展示平台风格的确认模态框
- **AND** 仅当管理员确认后系统才执行删除

#### Scenario: 招聘表单使用受控字段
- **WHEN** 管理员新增或编辑一条招聘岗位
- **THEN** 系统必须要求选择有效的所属企业、经验要求和薪资待遇
- **AND** 系统必须允许管理员通过多选方式维护专业要求、学历要求、学校类别和学校标签
- **AND** 系统不得要求管理员以自由文本或数值区间方式录入这些受控字段

#### Scenario: 招聘表单中的匹配条件允许为空
- **WHEN** 管理员新增或编辑一条招聘岗位时未填写专业要求、学历要求、学校类别或学校标签中的任一维度
- **THEN** 系统 MUST 允许岗位保存
- **AND** 系统 MUST 将该维度视为未配置匹配条件

#### Scenario: 招聘表单中的专业要求显式选择不限
- **WHEN** 管理员在招聘岗位表单中为“专业要求”选择“不限”
- **THEN** 系统 MUST 将该字段视为未配置专业要求
- **AND** 系统 MUST 不保存任何专业要求关系记录
- **AND** 系统 MUST 不允许“不限”与具体专业同时生效

#### Scenario: 招聘表单中的学历要求显式选择不限
- **WHEN** 管理员在招聘岗位表单中为“学历要求”选择“不限”
- **THEN** 系统 MUST 将该字段视为未配置学历要求
- **AND** 系统 MUST 不保存任何学历要求关系记录
- **AND** 系统 MUST 不允许“不限”与具体学历同时生效

#### Scenario: 编辑未配置要求的岗位时回显不限
- **WHEN** 管理员打开一条未配置专业要求或学历要求的招聘岗位编辑表单
- **THEN** 系统 MUST 将对应字段回显为已选择“不限”
- **AND** 系统 MUST 不将该空值状态回显成看似漏填的空白多选框

#### Scenario: 招聘表单使用三级工作地区
- **WHEN** 管理员新增或编辑一条招聘岗位并填写工作地区
- **THEN** 系统必须提供省、市、区（县）级联选择
- **AND** 系统必须仅保存工作地区的区域代码组合

#### Scenario: 招聘表单中的区域控件宽度与其他字段一致
- **WHEN** 管理员打开招聘岗位新增或编辑表单
- **THEN** 工作地区控件 MUST 与其他常规字段使用一致的列宽规则
- **AND** 该区域控件在桌面端不得明显长于相邻的普通输入框或下拉框

### Requirement: 招聘信息关联企业
系统 MUST 要求每条招聘岗位信息关联一条已存在的企业信息，并在展示和筛选时复用企业及其他受控基础数据；招聘列表必须以统一风格展示岗位与企业的关联信息，并与其他后台列表页保持一致的视觉风格。招聘列表筛选区 MUST 支持按专业要求承接和展示来自其他业务入口的自动筛选条件，包括数据分析页的招聘信息专业分析跳转。招聘列表中的“专业要求”“学历要求”筛选 MUST 明确区分“全部”和“不限”：`全部` 表示不施加该字段筛选条件，`不限` 表示仅返回该字段未配置要求的岗位。

#### Scenario: 录入岗位时选择企业
- **WHEN** 管理员新增或编辑招聘岗位信息
- **THEN** 系统要求选择一个有效企业，并保存该岗位与企业之间的关联关系

#### Scenario: 招聘列表展示企业信息
- **WHEN** 管理员查看招聘信息列表
- **THEN** 系统同时展示岗位关联的企业基础信息，满足日常管理需要

#### Scenario: 按受控字段筛选招聘岗位
- **WHEN** 管理员按所属企业、专业要求、学历要求、经验要求、薪资待遇或工作地区查询招聘岗位
- **THEN** 系统必须仅返回符合对应受控条件的岗位记录

#### Scenario: 按不限筛选专业要求为空的岗位
- **WHEN** 管理员在招聘列表筛选中将“专业要求”选择为“不限”
- **THEN** 系统 MUST 仅返回未配置任何专业要求的岗位
- **AND** 系统 MUST 不返回已配置具体专业要求的岗位

#### Scenario: 按不限筛选学历要求为空的岗位
- **WHEN** 管理员在招聘列表筛选中将“学历要求”选择为“不限”
- **THEN** 系统 MUST 仅返回未配置任何学历要求的岗位
- **AND** 系统 MUST 不返回已配置具体学历要求的岗位

#### Scenario: 按全部查看岗位时不施加专业或学历筛选
- **WHEN** 管理员在招聘列表筛选中保留“全部专业”或“全部学历”
- **THEN** 系统 MUST 不因该字段额外过滤招聘岗位
- **AND** 系统 MUST 将“全部”和“不限”视为不同的筛选语义

#### Scenario: 分析页跳转后显示专业筛选状态
- **WHEN** 管理员从数据分析页点击某个招聘专业并进入招聘信息列表页
- **THEN** 系统 MUST 自动按该专业对应的受控专业编码筛选岗位
- **AND** 系统 MUST 在招聘列表筛选区显式展示当前已生效的专业要求筛选状态

#### Scenario: 列表页可继续调整分析带入的专业筛选
- **WHEN** 管理员通过数据分析跳转进入已带专业筛选条件的招聘信息列表页
- **THEN** 系统 MUST 允许管理员在列表页继续修改、清空或重新组合专业要求筛选条件

## ADDED Requirements

### Requirement: 岗位匹配学生忽略学校标签条件
系统 MUST 在岗位匹配学生链路中仅使用岗位已填写的专业要求、学历要求和学校类别筛选学生，不得继续将学校标签作为匹配条件。

#### Scenario: 根据岗位条件匹配学生
- **WHEN** 管理员打开某个岗位的“匹配学生”结果
- **THEN** 系统 MUST 仅按该岗位已填写的专业要求、学历要求和学校类别筛选在校大学生
- **AND** 系统 MUST 忽略该岗位中未填写的匹配维度
- **AND** 系统 MUST 不得因岗位学校标签设置而额外过滤学生

### Requirement: 岗位匹配学生结果展示薪资待遇
系统 MUST 在岗位匹配学生结果页中以岗位属性“薪资待遇”替代原学校标签摘要位置，确保页面展示与当前匹配口径一致。

#### Scenario: 查看岗位匹配学生结果摘要
- **WHEN** 管理员打开某个岗位的“匹配学生”结果
- **THEN** 系统 MUST 展示岗位的专业要求、学历要求、学校类别和薪资待遇摘要
- **AND** 系统 MUST 不再展示学校标签摘要占位

### Requirement: 招聘岗位删除保留软删除语义
系统 MUST 继续将招聘岗位视为可恢复的业务内容数据。管理员删除招聘岗位时，系统 MUST 对岗位主记录执行软删除，并在默认招聘列表、岗位检索和分析查询中排除该岗位。

#### Scenario: 删除招聘岗位后从默认列表中移除
- **WHEN** 管理员删除一条招聘岗位
- **THEN** 系统 MUST 将岗位主记录标记为已删除而不是物理移除
- **AND** 该岗位 MUST 不再出现在默认招聘列表和默认查询结果中

#### Scenario: 删除招聘岗位后默认统计忽略已删除岗位
- **WHEN** 系统执行招聘岗位列表统计、匹配查询或专业分析查询
- **THEN** 系统 MUST 默认忽略已软删除的岗位主记录
- **AND** 这些岗位 MUST 不影响默认展示数量与统计结果

### Requirement: 招聘岗位关联关系使用物理删除
系统 MUST 将招聘岗位的专业要求、学历要求、学校类别和学校标签关系视为临时配置关系，而不是可恢复业务内容。系统删除或重建岗位匹配条件时，相关关系表 MUST 使用物理删除语义。

#### Scenario: 删除岗位时清理关联关系
- **WHEN** 管理员删除一条招聘岗位
- **THEN** 系统 MUST 物理删除该岗位在专业、学历、学校类别和学校标签关系表中的关联记录
- **AND** 系统 MUST 不保留这些已删除关系作为软删除历史

#### Scenario: 编辑岗位时重建关联关系
- **WHEN** 管理员更新一条招聘岗位的匹配条件
- **THEN** 系统 MUST 先物理删除旧的关系记录再写入新的关系记录
- **AND** 系统 MUST 不因为历史已删除关系而阻塞同一关系的重新创建

### Requirement: 招聘信息支持人工排序字段
系统 MUST 在招聘岗位维护链路中提供一个名为“排序”的可空整数型字段。管理员 MUST 在新增与编辑招聘岗位时填写或清空该字段，系统 MUST 在招聘岗位详情中展示当前排序值，并保持详情页只读不直接编辑该字段。

#### Scenario: 在新增或编辑时维护排序值
- **WHEN** 管理员在招聘岗位新增或编辑表单中填写一个整数排序值或留空提交
- **THEN** 系统 MUST 保存该排序值或空值
- **AND** 系统 MUST 允许该字段不填

#### Scenario: 在详情中查看排序值
- **WHEN** 管理员打开一条招聘岗位详情
- **THEN** 系统 MUST 展示该记录当前保存的“排序”值
- **AND** 系统 MUST 不在详情页提供直接编辑该字段的交互

### Requirement: 招聘信息列表按人工排序优先展示
系统 MUST 在招聘信息列表中优先按人工排序字段展示记录。已填写“排序”的记录 MUST 排在未填写记录之前，并按排序值升序排列；当多个记录的排序值相同时，系统 MUST 继续按 `id desc` 排序；未填写排序值的记录 MUST 继续按现有 `id desc` 规则排序。招聘信息列表页 MUST 不再显示“序号”列。

#### Scenario: 已填写排序值的记录优先显示
- **WHEN** 招聘信息列表同时存在已填写排序值和未填写排序值的记录
- **THEN** 系统 MUST 先展示所有已填写排序值的记录
- **AND** 这些记录 MUST 按排序值从小到大排列

#### Scenario: 排序值相同或为空时继续使用默认顺序
- **WHEN** 多条招聘岗位的排序值相同，或这些记录都未填写排序值
- **THEN** 系统 MUST 在对应分组内继续按 `id desc` 排列

#### Scenario: 列表页不再显示序号列
- **WHEN** 管理员查看招聘信息列表
- **THEN** 系统 MUST 不再渲染“序号”列
- **AND** 系统 MUST 保留现有分页信息与其他业务列
