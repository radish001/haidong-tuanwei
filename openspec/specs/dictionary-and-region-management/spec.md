## MODIFIED Requirements

### Requirement: 字典数据统一管理
系统 MUST 提供固定六大类的基础数据管理能力，用于维护青年、企业、招聘三类业务所需的受控字典、主数据和区域数据，并供页面录入、筛选、导入导出和统计分析统一使用。学校、专业、专业分类、学历和学位相关主数据 MUST 同时维护官方编码与名称，并保证编码唯一、名称受控。招聘岗位对专业和学历的多选引用 MUST 视为正式业务引用，并参与主数据删除校验。

#### Scenario: 打开基础数据管理页
- **WHEN** 管理员进入字典管理页面
- **THEN** 系统必须展示固定的六个顶层页签
- **AND** 系统不得提供新增或删除顶层大类的能力

#### Scenario: 切换六大类页签
- **WHEN** 管理员切换公共字典、专业类别、专业名称、学校类别、学校、企业字典等任一页签
- **THEN** 系统必须切换到对应的数据列表和操作区
- **AND** 系统必须保持与后台统一管理页一致的查询、分页和抽屉交互样式

#### Scenario: 维护公共字典
- **WHEN** 管理员进入公共字典页签
- **THEN** 系统必须仅允许维护性别、民族、政治面貌等固定公共字典
- **AND** 系统不得允许管理员创建新的公共字典大类

#### Scenario: 维护企业和招聘相关固定字典
- **WHEN** 管理员进入企业字典页签
- **THEN** 系统必须允许维护企业规模、企业性质、企业行业、经验要求、薪资待遇五类固定字典
- **AND** 系统不得允许管理员创建新的企业字典大类

#### Scenario: 维护学校主数据编码
- **WHEN** 管理员新增或编辑一条学校主数据
- **THEN** 系统 MUST 要求同时填写学校官方编码和学校名称
- **AND** 系统 MUST 保证学校官方编码在学校主数据范围内唯一
- **AND** 系统 MUST 允许后续业务模块通过学校官方编码关联学校类别和学校标签

#### Scenario: 维护专业主数据编码
- **WHEN** 管理员新增或编辑一条专业主数据
- **THEN** 系统 MUST 要求同时填写专业官方编码和专业名称
- **AND** 系统 MUST 保证专业官方编码在专业主数据范围内唯一
- **AND** 系统 MUST 允许后续业务模块通过专业官方编码关联专业类别

#### Scenario: 维护专业分类编码
- **WHEN** 管理员新增或编辑一条专业分类主数据
- **THEN** 系统 MUST 要求同时填写专业分类编码和专业分类名称
- **AND** 系统 MUST 保证专业分类编码在专业分类主数据范围内唯一
- **AND** 系统 MUST 允许专业主数据和青年业务通过专业分类编码关联该分类

#### Scenario: 维护学历编码
- **WHEN** 管理员新增或编辑一条学历主数据
- **THEN** 系统 MUST 要求同时填写学历编码和学历名称
- **AND** 系统 MUST 保证学历编码在学历主数据范围内唯一
- **AND** 系统 MUST 允许青年信息和招聘信息通过学历编码关联该主数据

#### Scenario: 维护学位编码
- **WHEN** 管理员新增或编辑一条学位主数据
- **THEN** 系统 MUST 要求同时填写学位编码和学位名称
- **AND** 系统 MUST 保证学位编码在学位主数据范围内唯一
- **AND** 系统 MUST 允许后续业务模块通过学位编码关联该主数据

#### Scenario: 学校或专业名称变更不影响业务编码关联
- **WHEN** 管理员修改学校名称或专业名称，但保留原官方编码
- **THEN** 系统 MUST 保持已有业务数据继续通过原官方编码关联该主数据
- **AND** 系统 MUST 不要求通过名称批量回写青年信息中的学校或专业标识

#### Scenario: 删除专业主数据前校验岗位引用
- **WHEN** 管理员删除一条已被招聘岗位专业要求引用的专业主数据
- **THEN** 系统 MUST 拒绝删除
- **AND** 系统 MUST 提示该专业已被招聘岗位使用

#### Scenario: 删除学历主数据前校验岗位引用
- **WHEN** 管理员删除一条已被招聘岗位学历要求引用的学历主数据
- **THEN** 系统 MUST 拒绝删除
- **AND** 系统 MUST 提示该学历已被招聘岗位使用

#### Scenario: 在公共字典范围内维护区域数据
- **WHEN** 管理员进入公共字典相关管理区域
- **THEN** 系统必须允许管理员查看和维护省、市、区（县）区域数据
- **AND** 系统不得因此新增新的顶层大类

#### Scenario: 删除字典项前校验关联使用
- **WHEN** 管理员删除一条已被青年信息、企业信息或招聘岗位引用的字典项
- **THEN** 系统必须拒绝删除
- **AND** 系统必须提示该字典项存在关联使用

### Requirement: 区域数据统一管理
系统 MUST 维护标准行政区域数据，并将 `sys_region` 作为青年、企业和招聘业务区域字段的统一来源，支持后台对省、市、区（县）进行级联增删改查和层级校验。

#### Scenario: 查看级联区域列表
- **WHEN** 管理员进入区域数据管理视图
- **THEN** 系统必须按省、市、区（县）层级展示区域数据
- **AND** 系统必须支持按父级区域查看对应子级区域

#### Scenario: 新增下级区域时选择父级
- **WHEN** 管理员新增一条市级或区县级区域数据
- **THEN** 系统必须要求管理员选择一个合法的上级区域
- **AND** 系统必须按区域层级保存父子关系

#### Scenario: 业务表单使用三级区域来源
- **WHEN** 青年信息、企业信息或招聘岗位表单需要录入区域
- **THEN** 系统必须从 `sys_region` 提供省、市、区（县）级联选项
- **AND** 系统必须按 `province_code`、`city_code`、`county_code` 三段口径供业务模块保存

#### Scenario: 区域选择支持部分层级
- **WHEN** 管理员在业务表单中只选择省级，或选择省级和市级但未选择区县
- **THEN** 系统必须允许保存已选择的上级区域代码
- **AND** 系统必须保证任何已选择的下级区域都隶属于已选择的上级区域

#### Scenario: 删除区域前校验下级区域
- **WHEN** 管理员删除一条仍存在下级区域的省级或市级区域数据
- **THEN** 系统必须拒绝删除
- **AND** 系统必须提示该区域存在下级区域关联

#### Scenario: 删除区域前校验业务引用
- **WHEN** 管理员删除一条已被青年信息、企业信息或招聘岗位使用的区域数据
- **THEN** 系统必须拒绝删除
- **AND** 系统必须提示该区域存在业务数据引用

## ADDED Requirements

### Requirement: 系统设置统一呈现基础数据工作台
系统 MUST 在左侧导航中继续保留“系统设置”作为一级菜单，并在进入该模块后统一呈现基础数据工作台；区域数据 MUST 作为工作台内部的固定管理内容使用，不得在当前页面层继续作为与字典管理并列强调的独立侧边子入口。

#### Scenario: 打开系统设置模块
- **WHEN** 管理员从左侧导航进入系统设置
- **THEN** 系统 MUST 展示统一的基础数据工作台页面
- **AND** 管理员能够在该页面内继续访问公共字典、专业、学校、企业字典和区域数据等管理内容

#### Scenario: 区域数据在工作台内维护
- **WHEN** 管理员需要维护区域数据
- **THEN** 系统 MUST 在系统设置工作台内部提供区域数据入口或页签
- **AND** 当前页面层不得将区域管理继续作为独立侧边子菜单重点呈现
