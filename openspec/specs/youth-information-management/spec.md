## MODIFIED Requirements

### Requirement: 青年信息分类统一管理
系统 MUST 使用统一的数据模型管理青年信息，并在数据层继续保留在校大学生、毕业未就业青年、农村社区待业青年、创业青年四类青年信息；但当前后台页面层 MUST 仅向管理员展示在校大学生视图，不得在青年信息库页面中展示其他青年分类入口、页签或说明文案。当前可见的青年信息页面顶部标签、筛选区、操作区和列表布局 MUST 保持统一页面结构，并与后台统一列表与抽屉交互规范一致。

#### Scenario: 打开青年信息库页面
- **WHEN** 管理员进入青年信息库模块
- **THEN** 系统 MUST 默认展示在校大学生列表
- **AND** 页面层不得展示毕业未就业、农村社区待业、创业青年等其他分类入口或页签

#### Scenario: 后端继续保留四类青年信息
- **WHEN** 系统数据库和后端路由仍保留四类青年信息的数据模型、存量数据或访问能力
- **THEN** 本次页面优化 MUST 不要求删除这些底层能力
- **AND** 当前后台页面仍 MUST 仅暴露在校大学生视图

#### Scenario: 在分类页面新增青年信息
- **WHEN** 管理员在当前可见的青年信息页面新增一条青年信息
- **THEN** 系统将该记录保存到统一青年信息表中
- **AND** 当前页面按在校大学生分类视图继续展示该记录

### Requirement: 青年信息查询与维护
系统 MUST 支持通过受控字段和区域条件查询青年信息，并在当前可见的在校大学生列表页提供与后台统一规范一致的分页、局部刷新、抽屉维护和删除确认交互；列表页与表单页 MUST 采用正式、精简、适合业务管理场景的展示文案。学校、专业、学历和学位相关查询与维护 MUST 基于主数据官方编码完成，但页面展示仍 MUST 提供业务可读的学校名称、专业名称、学历名称和学位名称；专业类别相关筛选、存储和展示 MUST 统一以字典值作为保存与查询口径，并以字典标签作为页面主要展示内容。系统同时 MUST 支持按招聘岗位派生出的专业、学历、学校类别和学校标签条件筛选在校大学生，并支持在校学生详情中按完全一致的专业、学历、学校类别和学校标签条件反向筛选招聘岗位。当前在校大学生列表 MUST 在“出生年月”后展示“民族”字段，并不得继续以“就业方向”作为该列表的固定展示列。当前青年信息维护表单中的籍贯、学校所在地和现居住地控件 MUST 与其他常规表单字段保持一致的列宽规则，不得长期以明显更长的整行控件破坏表单对齐。当前在校大学生列表页中的专业筛选 MUST 支持由专业类别驱动的级联选择。

#### Scenario: 按条件筛选青年信息
- **WHEN** 管理员使用籍贯、性别、民族、政治面貌、学历、学位、学校、专业名称、姓名等条件查询青年信息
- **THEN** 系统仅返回符合条件的青年记录
- **AND** 系统 MUST 允许管理员通过受控学校、专业、学历和学位选项发起查询，即使底层数据按编码存储
- **AND** 系统以统一视觉风格刷新当前在校大学生列表

#### Scenario: 专业类别驱动专业级联筛选
- **WHEN** 管理员在青年信息列表页先选择一个专业类别
- **THEN** 系统 MUST 将专业下拉选项限制为该专业类别下的专业
- **AND** 当专业类别为空时系统 MUST 允许专业下拉恢复为全部专业选项

#### Scenario: 青年列表使用统一管理布局
- **WHEN** 管理员进入青年信息列表页
- **THEN** 系统 MUST 在页面上方展示查询筛选区
- **AND** 系统 MUST 在查询区域下方右侧展示新增、删除等操作按钮
- **AND** 系统 MUST 在列表底部展示分页信息、页号选择和每页条数选择

#### Scenario: 青年列表展示民族字段
- **WHEN** 管理员查看在校大学生列表中的基础信息列
- **THEN** 系统 MUST 在“出生年月”后展示“民族”列
- **AND** 系统不得继续在该列表中以“就业方向”作为固定展示列

#### Scenario: 青年列表分页局部刷新
- **WHEN** 管理员执行查询、切换页号或调整每页条数
- **THEN** 系统 MUST 仅刷新青年信息列表区域及其分页区域
- **AND** 系统不得触发整页跳转刷新

#### Scenario: 青年详情与编辑通过右侧抽屉打开
- **WHEN** 管理员在青年信息列表中点击新增、详情或编辑
- **THEN** 系统 MUST 在当前页面右侧滑出抽屉面板承载对应内容
- **AND** 系统不得通过整页跳转进入独立详情页或编辑页

#### Scenario: 编辑已存在的青年信息
- **WHEN** 管理员在右侧抽屉中修改一条已保存的青年信息并提交合法内容
- **THEN** 系统保存修改结果
- **AND** 系统关闭抽屉并刷新当前青年列表数据

#### Scenario: 删除青年信息前要求确认
- **WHEN** 管理员删除一条青年信息或批量删除多条青年信息
- **THEN** 系统 MUST 先展示平台风格的确认模态框
- **AND** 仅当管理员确认后系统才执行删除

#### Scenario: 按岗位匹配条件筛选在校大学生
- **WHEN** 系统根据某条招聘岗位已填写的专业要求、学历要求、学校类别和学校标签查询匹配学生
- **THEN** 系统 MUST 仅返回符合所有已填写维度条件的在校大学生
- **AND** 系统 MUST 忽略该岗位中未填写的匹配维度
- **AND** 系统 MUST 在结果中回显可读的学校名称、专业名称、学历名称和专业类别

#### Scenario: 从学生详情抽屉进入匹配招聘结果
- **WHEN** 管理员在一名在校学生的详情抽屉中点击“匹配招聘信息”
- **THEN** 系统 MUST 在当前页面右侧打开招聘匹配结果抽屉
- **AND** 该结果抽屉 MUST 覆盖当前学生详情抽屉，而不是直接关闭详情后跳回列表

#### Scenario: 关闭匹配招聘结果后返回学生详情抽屉
- **WHEN** 管理员关闭由学生详情进入的招聘匹配结果抽屉
- **THEN** 系统 MUST 恢复原学生详情抽屉内容
- **AND** 系统 MUST 不得直接返回青年信息列表页

### Requirement: 青年信息使用受控字段和三级区域
系统 MUST 要求青年信息录入和编辑使用受控选择项维护性别、民族、政治面貌、学历、学位、学校和专业名称，并使用省、市、区（县）三级代码维护籍贯、现居住地和学校所在地；学校、专业、学历和学位在青年信息中的最终保存口径 MUST 为主数据官方编码，学校所在地只在青年记录中维护，不得反向依赖学校主数据。青年信息中的 `majorCategory` MUST 保存为专业类别字典值，并作为青年信息筛选与展示链路中的统一类别标识。

#### Scenario: 录入青年信息时选择受控字段
- **WHEN** 管理员新增或编辑一条青年信息
- **THEN** 系统必须对性别、民族、政治面貌、学历、学位、学校和专业名称提供受控选项
- **AND** 系统不得要求管理员以自由文本方式录入这些字段

#### Scenario: 保存青年信息时写入学校、专业、学历和学位编码
- **WHEN** 管理员选择一所学校、一个专业、一个学历和一个学位并保存青年信息
- **THEN** 系统 MUST 根据已选主数据写入对应的学校官方编码、专业官方编码、学历编码和学位编码
- **AND** 系统 MUST 使用这些编码作为青年信息与学校、专业、学历、学位主数据的关联依据

#### Scenario: 展示青年信息时回显学校、专业、学历和学位名称
- **WHEN** 管理员查看青年信息列表、详情或编辑表单
- **THEN** 系统 MUST 基于青年信息中保存的学校编码、专业编码、学历编码和学位编码回显对应的学校名称、专业名称、学历名称和学位名称
- **AND** 系统 MUST 不要求业务页面直接展示裸编码作为主要阅读内容

#### Scenario: 保存青年信息时写入专业类别字典值
- **WHEN** 管理员选择一个专业并保存青年信息
- **THEN** 系统 MUST 根据所选专业对应的主数据类别写入专业类别字典值到 `majorCategory`
- **AND** 系统 MUST 不以专业类别展示名称作为该字段的持久化值

#### Scenario: 展示青年信息时回显专业类别标签
- **WHEN** 管理员查看青年信息中的专业类别，或使用专业类别筛选青年信息
- **THEN** 系统 MUST 以青年信息中保存的专业类别字典值关联专业类别字典，并回显对应的类别标签
- **AND** 系统 MUST 不要求业务页面直接展示专业类别字典值作为主要阅读内容

#### Scenario: 录入青年信息时选择三级区域
- **WHEN** 管理员新增或编辑一条青年信息并填写籍贯、现居住地或学校所在地
- **THEN** 系统必须提供省、市、区（县）级联选择
- **AND** 系统必须仅保存对应的区域代码组合

#### Scenario: 青年区域支持部分层级保存
- **WHEN** 管理员在青年信息中只选择省级，或只选择省级和市级
- **THEN** 系统必须允许保存已选择的区域层级
- **AND** 系统必须保证任何已保存的市级或区县级代码都具有合法的上级区域关系

#### Scenario: 学校所在地独立于学校主数据维护
- **WHEN** 管理员在青年信息中选择学校并填写学校所在地
- **THEN** 系统必须允许学校和学校所在地分别维护
- **AND** 系统不得要求学校所在地必须来自学校主数据中的固定地点

#### Scenario: 青年表单中的区域控件宽度与其他字段一致
- **WHEN** 管理员打开青年信息新增或编辑表单
- **THEN** 籍贯、学校所在地和现居住地控件 MUST 与其他常规字段使用一致的列宽规则
- **AND** 这些区域控件在桌面端不得明显长于相邻的普通输入框或下拉框

## ADDED Requirements

### Requirement: 青年信息维护招考年份
系统 MUST 在青年信息维护链路中支持“招考年份”字段，并将其作为仅保存年份值的业务属性与青年信息统一存储。管理员在新增、编辑和查看青年信息时 MUST 能录入、回显和查看该字段，且系统 MUST 不要求使用完整日期格式代替年份。

#### Scenario: 新增或编辑青年信息时维护招考年份
- **WHEN** 管理员在青年信息表单中填写一个合法的 4 位招考年份并保存
- **THEN** 系统保存该招考年份到对应青年记录
- **AND** 系统继续按现有青年信息维护流程完成新增或编辑

#### Scenario: 查看已保存青年信息时回显招考年份
- **WHEN** 管理员打开一条已保存青年信息的详情或编辑表单
- **THEN** 系统 MUST 回显当前记录已保存的招考年份
- **AND** 系统 MUST 以年份值展示该字段，而不是完整日期
