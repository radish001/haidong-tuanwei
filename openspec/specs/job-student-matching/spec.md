## ADDED Requirements

### Requirement: 管理员可按招聘岗位发起学生匹配
系统 MUST 允许管理员在招聘信息列表页基于单条选中的招聘岗位发起学生匹配，并展示与该岗位条件对应的在校大学生结果列表。

#### Scenario: 从招聘列表进入匹配结果
- **WHEN** 管理员在招聘信息列表中选中一条招聘岗位并点击“匹配学生”
- **THEN** 系统 MUST 打开该岗位的学生匹配结果视图
- **AND** 系统 MUST 展示岗位名称、所属企业、已生效的匹配条件和匹配到的学生数量

#### Scenario: 未选中或多选岗位时禁止发起匹配
- **WHEN** 管理员未选择岗位，或同时选择了多条岗位后点击“匹配学生”
- **THEN** 系统 MUST 拒绝进入匹配结果视图
- **AND** 系统 MUST 提示管理员仅能选择一条招聘岗位进行匹配

#### Scenario: 匹配结果只包含在校大学生
- **WHEN** 系统根据岗位条件查询学生匹配结果
- **THEN** 系统 MUST 仅返回 `youth_type = 'college'` 的青年记录
- **AND** 系统 MUST 不返回其他青年分类的数据

### Requirement: 岗位匹配条件按显式规则生效
系统 MUST 仅对招聘岗位中已填写的专业要求、学历要求、学校类别、学校标签四类条件执行筛选；未填写的条件 MUST 被自动忽略。不同维度之间 MUST 按 AND 组合，同一维度内的多选值 MUST 按 OR 命中。

#### Scenario: 空条件不参与筛选
- **WHEN** 某条招聘岗位未填写专业要求、学历要求、学校类别或学校标签中的任一维度
- **THEN** 系统 MUST 在匹配查询中忽略该维度
- **AND** 系统 MUST 不因为该维度为空而筛掉学生

#### Scenario: 专业要求匹配具体专业
- **WHEN** 某条招聘岗位填写了一个或多个专业要求
- **THEN** 系统 MUST 基于学生保存的专业官方编码匹配具体专业
- **AND** 系统 MUST 不得仅按专业类别或自由文本描述替代专业匹配

#### Scenario: 学历要求按枚举命中
- **WHEN** 某条招聘岗位填写了一个或多个学历要求
- **THEN** 系统 MUST 仅匹配学历编码属于该岗位已选学历集合的学生
- **AND** 系统 MUST 不得将某个学历自动推导为更高学历也可命中

#### Scenario: 学校类别按多选命中
- **WHEN** 某条招聘岗位填写了一个或多个学校类别
- **THEN** 系统 MUST 基于学生学校关联的学校类别进行筛选
- **AND** 学生命中其中任一已选学校类别即可通过该维度

#### Scenario: 学校标签按任一标签命中
- **WHEN** 某条招聘岗位填写了一个或多个学校标签
- **THEN** 系统 MUST 基于学生学校关联的学校标签进行筛选
- **AND** 学生学校命中其中任一已选学校标签即可通过该维度

### Requirement: 管理员可按在校学生发起招聘匹配
系统 MUST 允许管理员在在校学生详情抽屉中基于单条学生记录发起招聘匹配，并展示与该学生条件对应的招聘岗位结果列表。

#### Scenario: 从学生详情进入匹配岗位结果
- **WHEN** 管理员在一名在校学生的详情抽屉中点击“匹配招聘信息”
- **THEN** 系统 MUST 打开该学生对应的招聘匹配结果抽屉
- **AND** 系统 MUST 展示学生姓名、已生效的匹配条件和匹配到的招聘岗位数量

#### Scenario: 匹配结果仅返回招聘岗位
- **WHEN** 系统根据学生条件查询匹配结果
- **THEN** 系统 MUST 仅返回未删除的招聘岗位记录
- **AND** 系统 MUST 不返回学生记录或其他业务实体作为匹配结果

### Requirement: 学生匹配岗位条件按显式规则生效
系统 MUST 仅对学生已具备的专业、学历、学校类别、学校标签四类条件执行岗位筛选；学生侧缺失的维度 MUST 被自动忽略。不同维度之间 MUST 按 AND 组合，同一维度内的多选值 MUST 按 OR 命中，并且匹配规则 MUST 与岗位匹配学生保持完全一致。

#### Scenario: 学生专业匹配岗位专业要求
- **WHEN** 当前学生保存了专业官方编码，且招聘岗位配置了一个或多个专业要求
- **THEN** 系统 MUST 仅返回专业要求集合中包含该学生专业编码的岗位
- **AND** 系统 MUST 不得仅按专业类别或自由文本描述替代专业匹配

#### Scenario: 学生学历匹配岗位学历要求
- **WHEN** 当前学生保存了学历编码，且招聘岗位配置了一个或多个学历要求
- **THEN** 系统 MUST 仅返回学历要求集合中包含该学生学历编码的岗位
- **AND** 系统 MUST 不得自动放宽为更高或更低学历均可命中

#### Scenario: 学生学校类别匹配岗位学校类别要求
- **WHEN** 当前学生关联学校主数据存在学校类别，且招聘岗位配置了一个或多个学校类别要求
- **THEN** 系统 MUST 基于学生学校关联的学校类别 ID 进行筛选
- **AND** 学生命中其中任一已选学校类别即可通过该维度

#### Scenario: 学生学校标签匹配岗位学校标签要求
- **WHEN** 当前学生关联学校主数据存在一个或多个学校标签，且招聘岗位配置了一个或多个学校标签要求
- **THEN** 系统 MUST 基于学生学校关联的标签集合与岗位标签集合是否存在交集进行筛选
- **AND** 学生命中其中任一已选学校标签即可通过该维度

#### Scenario: 学生缺失维度时忽略对应岗位条件
- **WHEN** 当前学生缺失专业、学历、学校类别或学校标签中的任一维度
- **THEN** 系统 MUST 在匹配查询中忽略该维度
- **AND** 系统 MUST 不因为该维度缺失而直接判定所有岗位都不匹配

### Requirement: 匹配结果列表继承人工排序规则
系统 MUST 让岗位匹配学生结果页与学生匹配岗位结果页继承对应业务实体的人工排序规则。已填写“排序”的记录 MUST 排在未填写记录之前，并按排序值升序排列；当多个记录的排序值相同时，系统 MUST 继续按 `id desc` 排序；未填写排序值的记录 MUST 继续按现有 `id desc` 规则排序。

#### Scenario: 岗位匹配学生结果按青年排序规则展示
- **WHEN** 管理员打开某个岗位的“匹配学生”结果
- **THEN** 系统 MUST 先展示已填写青年信息排序值的记录
- **AND** 这些记录 MUST 按排序值升序、相同值按 `id desc` 排列
- **AND** 未填写排序值的青年记录 MUST 继续按 `id desc` 排列并排在其后

#### Scenario: 学生匹配岗位结果按招聘排序规则展示
- **WHEN** 管理员在青年详情中打开“匹配招聘信息”结果
- **THEN** 系统 MUST 先展示已填写招聘岗位排序值的记录
- **AND** 这些记录 MUST 按排序值升序、相同值按 `id desc` 排列
- **AND** 未填写排序值的岗位记录 MUST 继续按 `id desc` 排列并排在其后

### Requirement: 匹配结果列表不再显示序号列
系统 MUST 在岗位匹配学生结果页与学生匹配岗位结果页移除“序号”列，避免分页位置被误解为业务排序编号。

#### Scenario: 查看岗位匹配学生结果时不显示序号列
- **WHEN** 管理员查看岗位匹配学生结果列表
- **THEN** 系统 MUST 不再渲染“序号”列
- **AND** 系统 MUST 保留现有分页信息与其他业务列

#### Scenario: 查看学生匹配岗位结果时不显示序号列
- **WHEN** 管理员查看学生匹配岗位结果列表
- **THEN** 系统 MUST 不再渲染“序号”列
- **AND** 系统 MUST 保留现有分页信息与其他业务列
