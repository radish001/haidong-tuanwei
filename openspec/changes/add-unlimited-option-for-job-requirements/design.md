## Context

当前招聘岗位中的“专业要求”和“学历要求”使用多选控件维护，管理员如果想表达“不限”，只能通过不选择任何项来间接表达。底层实现已经将空的专业/学历关系解释为“不参与该维度匹配”，因此现有匹配与筛选逻辑天然支持“不限”，但页面交互和筛选语义不够直观。

这次变更只针对招聘模块，重点是把“不限”从隐式空值提升为显式交互选项，同时保持当前数据库结构和匹配语义不变。另一个约束是列表筛选中已经存在“全部专业/全部学历”，因此必须明确区分“全部”与“不限”：

- `全部`：不对该字段施加筛选条件
- `不限`：仅筛选该字段未配置要求的岗位

## Goals / Non-Goals

**Goals:**

- 让招聘岗位新增、编辑表单中的专业要求和学历要求支持显式选择“不限”
- 保持“不限”与现有空关系存储语义兼容，不引入新的数据库字段或关系值
- 让编辑回显能把当前空的专业/学历要求明确呈现为“不限”
- 让招聘列表筛选支持“全部”与“不限”的语义分离
- 统一招聘列表、详情和匹配结果中空要求的展示文案

**Non-Goals:**

- 不修改岗位匹配学生、学生匹配岗位的核心匹配规则
- 不引入新的字典值、专业主数据编码或数据库结构
- 不调整学校类别、学校标签、经验要求、薪资待遇等其他筛选项的交互语义
- 不改变“空值表示不限”的底层持久化模式

## Decisions

### 1. 使用前端/后端约定的 sentinel 表达显式“不限”，落库时仍转为空

新增、编辑表单中的“专业要求”“学历要求”继续复用现有多选控件，但增加一个前端专用的“不限”选项。提交时如果选择了该选项，后端会在归一化逻辑中将其转为空列表，而不是尝试写入专业关系表或学历关系表。

这样做的原因：

- 兼容当前“空关系即不限”的存储和匹配语义
- 不需要修改数据库结构或主数据约束
- 历史数据天然兼容，不需要数据迁移

备选方案是把“不限”作为真实字典值或专业编码存储，但这会污染主数据约束，并迫使匹配 SQL 和关系表语义改造，因此不采用。

### 2. “不限”与具体专业/学历互斥，编辑回显时将空值映射为“不限”

在表单交互上，“不限”不能与具体项同时存在：

- 选择“不限”时清空其他已选项
- 选择具体项时取消“不限”

在编辑场景中，如果岗位当前没有任何专业要求或学历要求，表单应显式显示为已选中“不限”，而不是显示为空未选。

这样做的原因：

- 避免“不限 + 本科”这类自相矛盾状态
- 让编辑页准确表达当前岗位配置，而不是把空值误解为漏填

### 3. 列表筛选中的“全部”与“不限”分离为两种不同查询语义

招聘列表筛选中的“专业要求”“学历要求”继续保留当前的“全部”默认选项，同时新增“不限”：

- `全部`：不加该字段筛选条件
- `不限`：仅返回该字段未配置要求的岗位
- 具体值：按当前存在关系记录包含该值的规则筛选

实现上，筛选查询需要增加对“不限”的特殊分支：

- 专业不限：岗位在 `job_post_major_rel` 中不存在任何专业关系
- 学历不限：岗位在 `job_post_education_rel` 中不存在任何学历关系

这样做的原因：

- “全部”和“不限”在用户心智上不是一回事
- 只有明确区分，管理员才能筛出“要求为空”的岗位

### 4. 空要求的展示文案统一为“不限”

招聘列表、招聘详情、岗位匹配学生结果、学生匹配岗位结果中，凡是专业要求或学历要求为空的岗位，页面统一展示为“不限”。

这样做的原因：

- 与新增、编辑、筛选中的显式“不限”保持一致
- 减少“未限制”“空白”“未配置”等多种文案并存造成的理解偏差

## Risks / Trade-offs

- **[风险] 前端多选控件引入“不限”后，可能与现有多选插件行为冲突** → 通过显式互斥逻辑和编辑回显测试覆盖“只选不限”“不限切换为具体项”“具体项切换为不限”
- **[风险] 筛选中的“全部”和“不限”文案相近，用户仍可能混淆** → 在设计上保留“全部专业/全部学历”的现有文案，同时将“不限”仅用于“要求为空”的岗位
- **[风险] 历史岗位空值回显为“不限”后，可能与过去空白展示不一致** → 统一列表、详情和匹配结果文案，减少新旧界面语义不一致
- **[权衡] 方案保留了“空值表示不限”的底层语义，无法区分管理员是主动选择不限还是从未配置** → 本次以最小改动优先，不引入新的持久化状态

## Migration Plan

1. 更新招聘表单和筛选交互，增加“不限”选项与互斥逻辑
2. 在后端归一化和查询逻辑中识别“不限” sentinel，并映射到现有空值语义
3. 统一招聘列表、详情和匹配结果中的空要求展示文案
4. 补充测试，覆盖新增/编辑回显、筛选语义和匹配兼容性
5. 发布时无需数据库迁移；如果需要回滚，只需恢复前端选项和筛选分支，已有数据无需处理

## Open Questions

- “不限”在页面展示文案上是否全部统一为“不限”，还是在匹配摘要中继续保留“未限制”？当前设计建议统一为“不限”。
