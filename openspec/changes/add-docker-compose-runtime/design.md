## Context

当前项目主要通过本地 JDK、Maven 和环境变量直接执行 `./mvnw spring-boot:run` 启动，数据库连接依赖 `MYSQL_*` 环境变量，日志目录和企业营业执照上传目录也依赖宿主机文件系统。项目尚未提供 Dockerfile、Compose 编排文件或容器部署文档，因此在新环境中启动应用需要重复安装依赖、配置数据库连接并手工约定目录结构。

用户希望新增 `docker/` 目录，使执行 `docker compose up -d` 后可以一键拉起 Spring Boot 应用和 MySQL 运行环境，但数据库表结构与初始化数据仍由后续手工导入。也就是说，本次变更要解决的是“标准化运行环境”，而不是“数据库自动初始化”。

## Goals / Non-Goals

**Goals:**
- 提供标准化 `docker/` 目录，包含应用镜像构建文件和 Compose 编排文件。
- 让 `docker compose up -d` 能一次启动应用容器和 MySQL 容器。
- 让应用容器继续通过现有 `MYSQL_*` 环境变量模式连接数据库服务。
- 为日志目录、上传目录和 MySQL 数据目录提供持久化挂载约定。
- 在文档中明确说明数据库表结构与业务数据需要手工导入。

**Non-Goals:**
- 不在本次变更中实现数据库 schema 或种子数据自动导入。
- 不引入 Kubernetes、Helm 或其他超出 Compose 范围的部署方式。
- 不改造业务代码以适配多环境配置中心或密钥管理系统。
- 不保证空数据库状态下应用功能可直接可用，只保证运行环境可被拉起。

## Decisions

### 1. 使用 `docker/` 独立目录承载部署文件
- 决定: 将 `Dockerfile`、`docker-compose.yml`、环境变量示例和部署文档统一放在 `docker/` 目录下，而不是散落在仓库根目录。
- 原因: 用户已明确希望新增 `docker` 目录，且独立目录能把业务代码与部署资产分开，便于后续扩展多个部署文件。
- 备选方案:
  - 放在仓库根目录: 更符合部分社区惯例，但会让根目录更杂乱，也不符合用户当前偏好。

### 2. 应用镜像采用 Maven 多阶段构建
- 决定: 使用多阶段 Dockerfile，第一阶段进行 Maven 构建，第二阶段仅保留运行所需 JRE 与可执行 jar。
- 原因: 多阶段构建可以避免将 Maven、源码和构建缓存带入最终运行镜像，减少镜像体积并降低运行时复杂度。
- 备选方案:
  - 单阶段镜像内直接构建并运行: 配置简单，但镜像体积大、职责不清晰。
  - 完全依赖宿主机先打 jar 再复制: 对使用者流程要求更高，不符合“一条 compose 命令拉起环境”的目标。

### 3. Compose 编排仅负责拉起环境，不负责导入数据库内容
- 决定: `docker-compose.yml` 中包含 `app` 和 `mysql` 两个服务，但不挂载自动初始化 SQL，不配置应用层自动建表。
- 原因: 用户已明确数据库内容后续手工导入，本次变更重点是标准化运行环境，而不是改变数据库初始化流程。
- 备选方案:
  - 挂载 SQL 到 MySQL 初始化目录: 更接近真正一键可用，但与当前需求边界不一致。
  - 让 Spring Boot 自动执行 `schema.sql` / `data.sql`: 会改变现有启动行为，并加剧文档与运行时配置分叉。

### 4. 为服务就绪与持久化提供基础部署保障
- 决定: 在 Compose 中为 MySQL 配置健康检查，应用服务依赖 MySQL 健康状态启动；同时挂载 MySQL 数据卷、应用日志目录和上传目录。
- 原因: 这样可以降低“应用先起但数据库未就绪”的启动失败概率，也能保证容器重建后仍保留数据库文件、日志和上传文件。
- 备选方案:
  - 仅使用 `depends_on` 不做健康检查: 容易出现数据库尚未可连接时应用提前启动失败。
  - 不做卷挂载: 运行简单，但会造成数据和上传文件在容器重建后丢失。

## Risks / Trade-offs

- [空数据库下应用可能无法正常提供业务功能] -> 在 Compose 文档中明确写出“容器环境已拉起，但仍需手工导入数据库表结构和数据”。
- [Maven 多阶段构建导致首次镜像构建较慢] -> 这是换取标准化镜像和更小运行镜像的合理成本，可通过 Docker 缓存缓解。
- [宿主机 Docker Compose 版本差异影响健康检查依赖语法] -> 优先使用当前主流 Compose 语法，并在文档中注明推荐版本。
- [现有 README 关于自动执行 `schema.sql` / `data.sql` 的表述与实际配置不一致] -> 在本次部署文档中给出明确容器化边界，避免用户误以为 Compose 会自动导库。
