## ADDED Requirements

### Requirement: 应用日志文件按月归档并按大小切分
系统 MUST 将应用运行日志写入统一的文件日志体系，并以月为归档周期管理日志文件。系统 MUST 在单月日志文件达到预设大小阈值后自动拆分后续文件，而不是持续写入单个无限增长的日志文件。

#### Scenario: 应用启动后创建当月日志文件
- **WHEN** 应用在任意月份启动并开始输出日志
- **THEN** 系统将日志写入当前月份对应的日志文件
- **AND** 当前月份的日志文件命名 MUST 包含 `yyyy-MM` 形式的月份标识

#### Scenario: 单月日志超过阈值后自动切分
- **WHEN** 当前月份的日志文件达到预设大小阈值
- **THEN** 系统 MUST 自动将后续日志写入同月份的下一个分片文件
- **AND** 同月份多个分片文件 MUST 可通过递增序号区分

### Requirement: 关键业务操作必须记录摘要日志
系统 MUST 对认证、核心增删改、批量删除、导入导出、系统设置变更和分析配置变更等关键业务动作记录摘要日志，以支持排障和审计。摘要日志 MUST 记录操作类型、执行结果及必要的业务标识，但 MUST 避免输出密码、完整文件内容或其他敏感大对象。

#### Scenario: 执行关键业务写操作
- **WHEN** 管理员执行新增、更新、删除、批量删除或配置变更操作
- **THEN** 系统 MUST 记录对应的摘要业务日志
- **AND** 日志内容 MUST 至少包含操作名称、业务对象类型、关键标识或数量、执行结果

#### Scenario: 执行导入或导出操作
- **WHEN** 管理员执行导入或导出操作
- **THEN** 系统 MUST 记录该次操作的摘要日志
- **AND** 导入日志 MUST 包含成功数量、失败数量或失败原因摘要
- **AND** 系统 MUST 不记录上传文件的完整数据内容

### Requirement: SQL 日志必须纳入统一日志框架
系统 MUST 将数据库访问日志纳入统一的应用日志框架管理，不得继续依赖直接输出到标准输出的独立 SQL 打印方式。系统 MUST 允许通过统一日志配置控制 SQL 日志的级别与输出位置。

#### Scenario: 数据访问日志接入统一框架
- **WHEN** 系统输出 MyBatis SQL 相关日志
- **THEN** 这些日志 MUST 通过统一日志框架输出
- **AND** 系统 MUST 支持通过日志配置控制是否记录 SQL 明细

### Requirement: 日志故障不得阻断主业务流程
系统 MUST 将日志记录视为旁路能力。即使日志写入、文件滚动或日志归档过程中出现异常，系统仍 MUST 继续执行业务请求的核心逻辑，不得仅因日志处理失败而返回业务失败。

#### Scenario: 日志写入异常时继续执行业务
- **WHEN** 系统在处理业务请求期间发生日志写入或日志滚动异常
- **THEN** 系统 MUST 继续执行该请求对应的核心业务逻辑
- **AND** 系统 MUST 不因日志处理异常单独中断认证、查询、增删改、导入或导出流程

#### Scenario: 业务成功不依赖日志成功
- **WHEN** 某次业务操作本身已经满足既有业务成功条件
- **THEN** 系统 MUST 按业务成功结果返回响应
- **AND** 系统 MUST NOT 将日志记录结果作为业务事务提交、回滚或响应状态的前置条件
