## Context

当前青年、企业、招聘、政策四个业务模块都已经引入 `sort_order` 字段，并在详情抽屉与新增/编辑抽屉中直接渲染“排序”输入或只读展示。系统设置页目前已经承载基础数据工作台与个别专项配置，但仓库中没有通用 `sys_config` 配置中心，也没有全局特性开关抽象。

这次变更是一个跨模块的页面显示控制需求：同一个全局开关需要同时影响系统设置页本身、四个业务模块的详情模板、四个业务模块的表单模板，以及这些页面的公共模型注入方式。现有需求还隐含一个重要约束: 关闭显示只是隐藏 UI，不是停用排序逻辑，因此不能破坏已有 `sort_order` 数据，也不能意外改变列表排序规则。

## Goals / Non-Goals

**Goals:**
- 在系统设置工作台提供一个全局可持久化的“是否显示排序设置”开关。
- 让青年、企业、招聘、政策的详情与新增/编辑抽屉统一根据该开关决定是否展示“排序”字段。
- 在开关关闭时保持已有 `sort_order` 数据和列表排序行为不变。
- 明确关闭开关后的保存语义，避免编辑已有记录时因表单缺失字段而把排序值误清空。
- 让该配置以项目当前风格落地，尽量复用已有系统设置与全局模型注入机制。

**Non-Goals:**
- 不调整青年、企业、招聘、政策列表及匹配结果页的排序 SQL 规则。
- 不引入拖拽排序、批量排序、列表内联排序或新的排序算法。
- 不一次性抽象出通用的动态配置平台、任意键值配置中心或完整 feature flag 框架。
- 不改变 `sort_order` 字段的数据结构、数据库列类型或历史数据内容。

## Decisions

### 1. 使用专用的全局配置项，而不是构建通用配置中心

本仓库当前没有通用系统配置表或键值配置框架，系统设置页的现有能力也偏向“明确场景、明确存储”的专项配置。因此本次采用一个专用的布尔配置项来承载“是否显示排序设置”，而不是为了单个开关引入通用 `sys_config` 抽象。

这样做的好处是实现面更小，能延续当前代码库“显式配置、显式查询”的风格，也更容易在后续审阅时确认数据来源与影响范围。

备选方案：
- 新建通用配置表和统一配置服务：抽象能力更强，但对当前范围明显过重。
- 直接写死在 `application.yml`：缺少后台可维护性，不符合“在系统设置中做开关”的需求。

### 2. 将开关默认值设为“开启”，保持与当前线上展示兼容

当前所有相关页面都默认显示“排序”字段，因此新配置的默认值必须等价于“开启”，才能保证部署后在管理员未主动修改配置前页面行为不发生变化。

备选方案：
- 默认关闭：会让现有页面行为在上线后立刻变化，属于不必要的破坏性切换。

### 3. 通过全局页面模型注入配置值，避免每个业务 Controller 重复传参

项目中已经通过 `AdminViewModelAdvice` 给所有后台页面统一注入公共模型。排序显示开关同样适合作为全局页面属性注入，例如 `sortFieldVisible`，这样青年、企业、招聘、政策模板都可以直接复用该变量，而不需要在多个 Controller 的详情和表单方法中逐一 `model.addAttribute(...)`。

备选方案：
- 每个 Controller 单独查询并注入：实现分散，后续容易漏改。
- 前端通过单独接口再拉取配置：增加一次异步状态管理，对当前 Thymeleaf 页面模式没有必要。

### 4. 关闭显示只影响 UI，可编辑记录的既有排序值必须保留

当开关关闭后，编辑表单不再渲染 `sortOrder` 字段。如果后端仍沿用“表单里没有该字段就按 `null` 回写”的逻辑，会导致管理员编辑其他字段时把原排序值意外清空。因此需要在设计上明确：

1. 开关关闭时，详情页和表单页都不展示“排序”字段。
2. 新增记录时不要求前端提交排序值，未提交时保持空值即可。
3. 编辑已有记录时，如当前请求未包含排序字段且开关关闭，系统应保留原记录的 `sort_order`，而不是回写 `null`。

备选方案：
- 关闭时继续在表单中保留隐藏字段：可以保值，但语义上仍是前端参与维护，不如后端显式保护清晰。
- 关闭时统一清空排序值：会把“隐藏显示”误做成“关闭排序能力”，与需求相悖。

### 5. 系统设置页继续放在现有基础数据工作台内，而不是新增独立菜单

当前系统设置已经被规范为统一基础数据工作台，本次开关属于同一工作台内的系统级显示配置。设计上继续在现有页面内增加一个固定配置区或页签，避免新增侧边菜单、独立路由层级或新的设置子系统。

备选方案：
- 单独新增一个“显示设置”一级菜单：信息架构变动大，不符合当前系统设置的组织方式。

## Risks / Trade-offs

- [隐藏表单字段后编辑请求误清空旧排序值] → 在设计与规格中明确“关闭开关时保留既有排序值”，实现时在服务层或映射层兜底。
- [全局模型注入会让更多页面拿到该变量] → 接受这类低成本扩散，换取模板接入简单且一致。
- [专用配置实现未来可能出现第二、第三个类似开关] → 当前先选择最小方案；若后续系统设置项明显增多，再以已有专用配置为事实基础抽象通用层。
- [管理员可能误以为关闭开关后列表不再按排序展示] → 在系统设置文案中明确说明该开关只控制页面是否显示排序字段，不影响当前排序结果。

## Migration Plan

1. 增加“是否显示排序设置”的持久化存储，并将默认值初始化为开启。
2. 在系统设置页提供开关入口和保存动作。
3. 在全局页面模型中注入该配置值。
4. 在青年、企业、招聘、政策的详情和表单模板中接入条件渲染。
5. 调整编辑保存链路，确保关闭开关后不会覆盖历史排序值。
6. 发布后如需回滚，可保留配置数据不使用；模板与后端逻辑回退后，默认行为仍等同于始终显示。
