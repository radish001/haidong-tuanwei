## MODIFIED Requirements

### Requirement: 后台统一视觉壳子
系统必须为后台管理端提供统一的驾驶舱风格页面壳子，覆盖侧边导航、顶部标题区、内容面板和全局主题配色，使管理员在不同模块间切换时保持一致的视觉体验；该页面壳子 MUST 优先基于 Bootstrap 的导航、标题、面板和抽屉组件语义构建，不得继续依赖一套与 Bootstrap 并行的自定义通用结构组件。

#### Scenario: 打开任一后台业务页面
- **WHEN** 已登录管理员进入首页、青年信息、企业信息、招聘信息、政策管理或分析页面
- **THEN** 系统展示统一的深色驾驶舱风格页面壳子，包括左侧导航、顶部标题区、统一面板边框和一致的页面背景层次
- **AND** 页面壳子的主结构 MUST 优先使用 Bootstrap 组件语义与 utility class 组合实现

#### Scenario: 打开带详情或表单侧边容器的后台页面
- **WHEN** 管理员在列表页打开详情、编辑或新建入口
- **THEN** 系统 MUST 使用 Bootstrap 提供的抽屉或等价官方组件行为承载侧边面板交互
- **AND** 系统不得继续依赖项目私有的通用抽屉结构和样式体系作为默认实现

### Requirement: 后台通用组件风格一致
系统必须对按钮、筛选栏、输入框、表格、提示信息和操作区采用统一的视觉规范，避免不同业务页面之间出现明显割裂的样式表现；在这些通用组件具备 Bootstrap 或已选插件能力的场景下，系统 MUST 优先使用 Bootstrap 原生组件以及 Tom Select、TreeselectJS 等已选插件，而不是继续保留并行的自定义组件实现。

#### Scenario: 查看不同模块的列表和表单页面
- **WHEN** 管理员在不同业务模块之间切换并查看列表页、表单页或详情页
- **THEN** 系统以统一的组件风格呈现筛选区、操作按钮、表格、表单控件和提示信息，并保持一致的间距、描边、高亮和交互反馈
- **AND** 这些通用组件 MUST 优先基于 Bootstrap 原生组件和已选插件实现

#### Scenario: 页面存在确认弹窗或增强选择控件
- **WHEN** 页面需要展示删除确认、批量操作确认、单选增强、多选增强或区域级联选择
- **THEN** 系统 MUST 复用 Bootstrap `modal` 与已选插件提供的组件能力
- **AND** 系统不得继续保留同类自定义确认弹窗、旧多选控件或旧区域级联控件作为默认实现

## ADDED Requirements

### Requirement: 主题覆盖文件仅承担主题覆盖职责
系统 MUST 将后台主题样式文件限制为 Bootstrap 和已选插件的主题覆盖层，用于统一配色、边框、阴影、状态色和必要视觉 token；`app.css` 不得继续承担后台通用布局、抽屉、弹窗、分页、多选或级联组件的独立结构实现。

#### Scenario: 检查后台主题样式文件
- **WHEN** 开发者检查后台全局主题样式文件和共享样式入口
- **THEN** `app.css` MUST 主要承载 Bootstrap、Tom Select 和 TreeselectJS 的主题覆盖
- **AND** 文件中不得继续维护通用抽屉、确认弹窗、旧多选或旧级联组件的完整结构样式实现

### Requirement: 被替代的旧样式和脚本必须删除
系统 MUST 在 Bootstrap 或已选插件完成替换后删除被替代的旧样式、旧脚本和仅用于兼容旧样式体系的冗余模板包装层，避免仓库中长期并存两套实现。

#### Scenario: 页面完成 Bootstrap 化收敛
- **WHEN** 某个后台页面或共享片段已经切换到 Bootstrap 和已选插件的正式实现
- **THEN** 系统 MUST 删除对应的旧样式、旧脚本和不再使用的兼容类名或包装结构
- **AND** 仓库中不得保留与正式实现并行的历史参考实现或死代码
