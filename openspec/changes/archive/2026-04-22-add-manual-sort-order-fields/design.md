## Context

当前青年信息、企业信息、招聘信息和政策信息的主列表都直接按 `id desc` 返回，青年匹配招聘与岗位匹配学生结果页也复用了这些默认查询行为。管理员无法通过业务优先级主动置顶重点记录，且列表中的“序号”列只是分页位置，不再适合作为排序语义展示。

这个变更会同时触及四张业务表、四套表单与详情页、四个主列表查询，以及两个匹配结果列表，属于跨模块的数据模型与展示规则调整。项目当前默认不会自动执行 `schema.sql`，因此除了代码修改外，还需要明确实际环境的手工数据库变更路径。

## Goals / Non-Goals

**Goals:**
- 为青年信息、企业信息、招聘信息和政策信息统一增加可空的整数型“排序”字段。
- 让所有相关列表优先按人工排序值升序展示，未填写排序值时继续沿用现有 `id desc` 兜底规则。
- 保持详情页只读，在新增/编辑表单中维护排序值，并在详情页回显该字段。
- 移除主列表与匹配结果列表中的“序号”列，避免与人工排序含义冲突。
- 让匹配结果页与主列表共享一致的排序口径，避免不同入口展示顺序不一致。

**Non-Goals:**
- 不引入拖拽排序、批量排序、列表页内联编辑排序或自动重排能力。
- 不改变现有筛选条件、分页行为、匹配条件逻辑或软删除语义。
- 不为该字段增加单独的筛选项、导入导出支持或统计分析口径。

## Decisions

### 1. 统一使用可空整数列 `sort_order`

四张业务表分别增加一个可空 `int` 列 `sort_order`，Java 层统一映射为 `Integer sortOrder`。选择可空整型而不是字符串或布尔置顶标记，是因为需求本质是“人工优先级排序”，需要支持多条记录之间的相对顺序，同时允许管理员不填写该字段以保留默认展示逻辑。

备选方案：
- 布尔型“是否置顶”：只能区分置顶/非置顶，无法表达多条记录之间的稳定先后关系。
- 用创建时间或发布时间偏移模拟排序：会污染业务字段语义，也不利于后续维护。

### 2. 列表统一采用“有值优先、升序优先、默认兜底”的 SQL 排序

所有受影响列表统一改为如下排序语义：

1. `sort_order` 非空记录优先
2. `sort_order` 按升序排列，数值越小越靠前
3. 相同 `sort_order` 的记录按 `id desc`
4. `sort_order` 为空的记录排在最后，并继续按 `id desc`

实现上直接在各 MyBatis 查询中显式写出排序表达式，而不是抽象成额外排序参数。这样可以保持行为稳定，且当前代码库本来就是在 XML 中固定排序。

备选方案：
- 让前端传排序参数：会扩大接口和控制器变更面，也不符合“统一排序规则”的固定业务需求。
- 把空值当作 `0`：会让未填写记录意外排到最前面，与“未填写时沿用当前规则”冲突。

### 3. 排序值只在新增/编辑表单中维护，详情页只展示

项目现有交互是“详情只读，编辑可改”，本次沿用该模式。四个模块的详情页展示排序值，新增/编辑表单提供一个普通数字输入框，允许为空，不在详情页直接修改。这样能避免引入新的交互模式，也与用户已确认的需求一致。

备选方案：
- 在详情页直接可编辑：需要改造详情抽屉语义和表单提交流程，超出当前范围。
- 列表页内联编辑：实现复杂度更高，也不符合现有后台页面模式。

### 4. 匹配结果页复用同一排序口径

青年匹配招聘和岗位匹配学生页面本质上也是业务列表，且底层已经复用对应 `search(...)` 查询。设计上不为匹配结果单独发明排序规则，而是明确把它们纳入同一人工排序口径，保证同一条记录在主列表与匹配结果中的先后关系一致。

备选方案：
- 匹配结果继续按旧规则：会导致用户在不同入口看到同一批记录顺序不一致，降低可预期性。

### 5. 移除受影响列表中的“序号”列

在启用人工排序后，分页序号只代表“当前页第几行”，不再具有业务排序含义。设计上统一删除主列表和匹配结果列表中的序号列，避免管理员误将其理解为真实排序编号。分页信息本身保留，不影响当前页码感知。

## Risks / Trade-offs

- [数据库变更需要手工执行] → 在任务中明确同步 `schema.sql` 与环境变更脚本/SQL，部署前先执行字段新增。
- [多个模块各自维护查询与表单，易出现漏改] → 采用模块清单式任务拆分，覆盖实体、DTO、Service、Controller、XML、模板六类位置。
- [空值排序表达式在不同查询里重复书写] → 接受少量重复，换取查询行为显式且易审阅；本次不引入额外 SQL 片段抽象。
- [去掉序号列后用户失去“当前页第几条”的快速定位] → 保留分页元信息，并通过稳定的人工排序字段表达真正的业务优先级。

## Migration Plan

1. 在 `schema.sql` 为 `youth_info`、`enterprise_info`、`job_post`、`policy_article` 新增可空 `sort_order` 字段。
2. 在实际环境手工执行对应 `ALTER TABLE ... ADD COLUMN sort_order INT NULL` 语句。
3. 部署应用代码，使实体映射、表单、详情页和查询排序规则同步生效。
4. 如需回滚代码，可保留数据库字段不使用；因为新字段可空且默认不参与旧逻辑，不需要先删列。
