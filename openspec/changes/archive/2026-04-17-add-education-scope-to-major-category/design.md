## Context

### 当前状态
- 专业类别存储在 `sys_dict_item` 表中，`dict_type='major_category'`
- 现有14个学科门类（哲学、经济学、法学、教育学、文学、历史学、理学、工学、农学、医学、管理学、艺术学、军事学、交叉学科）
- 数据分析页通过 `sys_dict_item` 关联 `sys_major_catalog` 再关联 `youth_info` 进行专业类别分布统计
- 系统设置页使用通用字典表单维护专业类别，只有"字典值"和"显示名称"两个字段

### 业务约束
- 本科专业有14大类，专科专业有19大类（部分类别名称相同但编码不同）
- 同一专业类别可能同时属于多个学历层次（如"工学"在本科和专科都存在）
- 研究生专业类别体系与本科基本一致
- 分析页面需要分别展示本科和专科的专业类别分布，不能混在一起

## Goals / Non-Goals

**Goals:**
- 专业类别支持配置所属学历层次（可多选），在维护层面实现本科/专科/研究生的分层管理
- 数据分析页独立展示"本科专业类别分布"和"专科专业类别分布"两张图表
- 保持现有专业类别编码体系不变，确保历史数据兼容性
- 系统设置页专业类别管理增加"所属学历层次"字段维护

**Non-Goals:**
- 不新建独立的专业类别表，继续使用 `sys_dict_item`
- 不导入专科专业目录主数据（本次只改类别配置，不改专业主数据）
- 不展示研究生专业类别分析图（本次只做维护层面预留）
- 不改变现有通用字典管理能力（其他字典类型不受此字段影响）

## Decisions

### Decision 1: 字段设计方案
**选择**：在 `sys_dict_item` 新增 `education_scopes` varchar(100) 字段

**理由**：
- 改动量最小，不需要新建表、不需要改外键关系
- `sys_major_catalog` 通过 `category_dict_item_id` 关联的是字典项ID，这个关系保持不变
- 只需要改查询时的过滤条件，不需要改关联逻辑

**其他方案考虑**：
- 独立建表 `sys_major_category`：更规范，但改动量翻倍（需要改实体、改DAO、改Service、改页面、迁移数据）
- 用字典扩展表存KV：过度设计，且查询需要JOIN，增加复杂度

### Decision 2: 学历层次值设计
**选择**：代码中定义枚举常量，字段存储逗号分隔的常量名

```java
public enum MajorCategoryEducationScope {
    JUNIOR_COLLEGE("专科专业"),
    UNDERGRADUATE("本科专业"),
    GRADUATE("研究生专业");
}
```

**存储示例**：
- 只属本科：`"UNDERGRADUATE"`
- 同属本专科：`"UNDERGRADUATE,JUNIOR_COLLEGE"`

**理由**：
- 选项固定，不需要走字典表配置
- 存储可读性较好，方便直接查看数据库
- SQL查询时可用 `LIKE '%UNDERGRADUATE%'` 进行过滤

**其他方案考虑**：
- 存位掩码（bitmask）：节省空间但可读性差，SQL查询不直观
- 存JSON数组：查询需要JSON函数支持，增加数据库依赖
- 走关联表：需要新建表，用户明确要求不加表

### Decision 3: 数据分析口径设计
**选择**：本科/专科图分别查询，按专业类别的 `education_scopes` 过滤

**查询逻辑**：
- 本科专业类别分布：查询 `education_scopes` 包含 `UNDERGRADUATE` 的类别
- 专科专业类别分布：查询 `education_scopes` 包含 `JUNIOR_COLLEGE` 的类别
- 同时结合学生的 `education_level` 字段进行交叉验证（本科图只统计学历为本科的学生）

**理由**：
- 满足业务需求：分别展示本科和专科的专业类别分布
- 口径一致：用专业类别配置的层次过滤，同时用学生实际学历验证

### Decision 4: 历史数据迁移策略
**选择**：现有14个专业类别默认设为 `"UNDERGRADUATE"`

**理由**：
- 保持现有分析口径：现在的图展示的就是本科专业类别分布
- 平滑过渡：改完后原图数据不会变，只是标题明确为"本科专业类别分布"

## Risks / Trade-offs

### Risk 1: 专科图可能没有数据
**问题**：当前 `sys_major_catalog` 里只有本科专业目录，专科学生的 `major_code` 可能无法关联到类别

**影响**：专科专业类别分布图可能为空或数据不准确

**缓解**：
- 明确告知业务方需要补充专科专业目录才能看到准确的专科分析图
- 本次变更只负责"按学历层次分类"的能力，不负责专科专业主数据导入

### Risk 2: `sys_dict_item` 新增字段影响其他字典类型
**问题**：`education_scopes` 只对 `major_category` 有意义，其他字典类型（性别、民族等）不应该有这个字段

**影响**：表结构上有这个字段，但业务上其他类型不需要使用

**缓解**：
- 在代码层面控制，只有 `dict_type='major_category'` 时才处理 `education_scopes`
- 其他字典类型的表单不展示这个字段
- 数据库字段允许为空，其他类型不填即可

### Risk 3: LIKE 查询性能问题
**问题**：用 `LIKE '%UNDERGRADUATE%'` 过滤可能无法使用索引

**影响**：如果专业类别数量很大（实际只有14个，风险很小），分析查询可能变慢

**缓解**：
- 专业类别数量固定且很少（14-33个），全表扫描也不影响性能
- 如需优化，可改为存位掩码用位运算，但当前场景不需要

## Migration Plan

### 数据库迁移
```sql
-- 1. 新增字段
ALTER TABLE sys_dict_item ADD COLUMN education_scopes VARCHAR(100) NULL COMMENT '所属学历层次（专科专业/本科专业/研究生专业）';

-- 2. 更新现有专业类别数据
UPDATE sys_dict_item 
SET education_scopes = 'UNDERGRADUATE' 
WHERE dict_type = 'major_category';
```

### 部署步骤
1. 执行数据库 DDL
2. 部署新版本代码（向后兼容，旧代码忽略新字段）
3. 验证专业类别管理页面正常
4. 验证数据分析页面图表正常

### 回滚策略
- 数据库字段可保留，旧代码不读取该字段，不影响运行
- 如需回滚代码，数据库字段可以保留（不删除），下次升级继续使用
