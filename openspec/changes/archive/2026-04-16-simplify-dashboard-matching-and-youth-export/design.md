## Context

本次变更跨越首页驾驶舱、学生与岗位匹配链路以及学生信息 Excel 导出链路，但都属于已有功能的规则收敛与展示简化，而不是新增业务模块。当前系统在三个方面与最新业务口径不一致：首页仍展示两块业务摘要表格；双向匹配仍把学校标签作为硬性条件；学生导出表头与导入模板顺序不一致，业务人员难以直接复用导出结果。

这些改动主要落在现有控制器、模板、查询条件和导出列定义上，不涉及数据库结构调整，也不改变招聘岗位录入时对学校标签的维护能力。

## Goals / Non-Goals

**Goals:**
- 精简首页，只保留当前需要的核心统计和地图展示，不再展示政策信息与招聘信息摘要表格。
- 将学生与岗位的双向匹配条件统一收敛为专业、学历和学校类别，移除学校标签参与匹配的行为。
- 让匹配结果页展示与新规则一致：岗位匹配学生页显示薪资待遇，学生匹配岗位页不再显示学校标签占位。
- 让学生信息导出列顺序与导入模板完全一致，降低导入导出之间的人工转换成本。

**Non-Goals:**
- 不移除招聘岗位表单中的学校标签维护能力，也不修改学校标签主数据本身。
- 不调整首页统计卡片中的总量口径，不移除岗位总数或政策总数卡片。
- 不扩展学生导入模板字段范围，不新增“专业类别”等导入列。
- 不引入新的匹配评分、模糊匹配或权重排序逻辑。

## Decisions

### 1. 首页仅删除摘要区块，同时保留现有统计与地图数据结构

首页摘要区块是纯展示层收缩，优先删除模板中的政策/招聘摘要表格与入口，同时移除控制器中对应的摘要数据加载。这样可以最小化影响范围，不改动已有统计卡片、地图聚合和分析跳转行为。

备选方案是保留摘要接口但隐藏前端展示。该方案实现更快，但会继续保留无用查询与模型数据，长期会增加维护噪音，因此不采用。

### 2. 双向匹配都在控制器层停止下发学校标签条件，并同步移除 SQL 条件依赖

当前双向匹配分别由控制器组装查询对象，再由 MyBatis XML 根据 `schoolTagIds` 拼接 `exists` 条件。新的业务口径是不再使用学校标签作为匹配维度，因此同时在控制器层取消设置 `schoolTagIds`，并在对应查询 SQL 中去掉匹配链路上的学校标签过滤分支。

仅在控制器层把 `schoolTagIds` 设为空虽然可以达到部分效果，但仍会保留无效 SQL 分支与概念负担；仅删 SQL 而保留控制器赋值又会让查询对象语义与实际行为不一致，因此两层同时调整。

### 3. 双向匹配结果页分别按各自业务语境收缩展示

`岗位匹配学生` 页面顶部摘要中，原“学校标签”位置改为展示岗位已有受控字段“薪资待遇”，用于补足页面可读信息且不引入新的匹配含义。`学生匹配招聘信息` 页面则直接去掉“学校标签”展示位，不做字段替换，避免将非匹配字段误解为筛选依据。

备选方案是两边都统一替换为“薪资待遇”。该方案视觉更对称，但学生匹配岗位页顶部区域本身是在回显学生参与匹配的条件，而薪资待遇属于岗位属性，不适合作为学生侧条件摘要，因此不采用。

### 4. 学生导出以导入模板为单一字段顺序基准

学生导出表头和写列顺序统一对齐到导入模板当前定义，确保业务人员可以直接基于导出结果整理后再导入。由于导入模板没有“专业类别”列，导出中该列同步移除，避免形成额外结构差异。

备选方案是保留“专业类别”并只调整其余列顺序。该方案仍无法做到“导入导出一致”，会继续让导出文件多出一列，因此不采用。

## Risks / Trade-offs

- [学校标签仍可录入但不参与匹配，容易让维护人员误解字段用途] → 在规格和后续实现中明确学校标签只保留录入与展示用途，不作为当前学生岗位匹配条件。
- [移除首页摘要后，用户可能失去最近政策和岗位的快速浏览入口] → 保留首页统计卡片和原业务列表入口，确保仍可从导航或统计卡片进入详情列表。
- [导出移除“专业类别”后，个别依赖旧列顺序的人工表格可能受影响] → 在变更说明中明确导出现在以导入模板为准，后续如确有需要再单独设计扩展导出版本。
- [只修改匹配结果页顶部摘要而不修改更深层详情文案，可能残留旧术语] → 实施时检查两个匹配模板中的列头和摘要标题，确保不再暗示学校标签参与匹配。
