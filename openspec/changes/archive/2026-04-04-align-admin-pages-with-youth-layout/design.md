## Context

当前项目已经把青年信息库打造成后台管理页的目标样式，包括查询区域、右侧操作区、分页、局部刷新、删除确认模态框和右侧抽屉交互。但企业信息、招聘信息、就业创业政策等模块仍保留不同的布局习惯与页面跳转方式，导致同一后台中存在多套交互模型。

这次变更是一个跨模块的前端与控制器协同改造，涉及：
- 多个列表模板的统一布局调整
- 控制器对整页渲染与 Ajax 片段渲染的双模式支持
- 抽屉脚本、列表局部刷新脚本、确认模态框脚本的统一复用
- 侧边栏菜单文案与分组逻辑的统一

约束条件：
- 不新增后端实体或数据库结构
- 不改变现有核心 CRUD 路由语义
- 保持现有表单校验、导入导出、分页与删除能力可继续工作
- 优先复用青年信息库已经验证过的页面结构和交互组件

## Goals / Non-Goals

**Goals:**
- 让青年信息、企业信息、招聘信息、就业创业政策的列表页在视觉结构和交互路径上保持一致
- 让新增、编辑、详情统一通过右侧抽屉承载，减少整页跳转
- 让列表查询、分页和刷新统一通过 Ajax 局部更新完成
- 让删除确认统一为平台风格模态框
- 让侧边栏导航文案与入口表现保持一致，降低认知负担

**Non-Goals:**
- 不重构底层数据模型或 DAO 查询逻辑
- 不引入前端框架或新的 UI 依赖
- 不为当前没有详情页的模块额外新增复杂业务字段或独立流程
- 不重新设计 dashboard、登录页或分析页布局

## Decisions

### 1. 以青年信息库作为统一布局基线

所有管理列表页统一采用“顶部查询区 + 查询区下方右侧操作区 + 列表区 + 分页区”的结构，避免每个模块维护一套独立布局。

选择原因：
- 青年信息库已经完成多轮视觉与交互优化，并且经过实际用户反馈
- 复用现有结构可以减少样式和模板分叉

替代方案：
- 为每个模块单独设计最佳布局
  - 放弃原因：虽然局部可更贴合业务，但会继续放大后台交互不一致的问题

### 2. 控制器采用“整页 + 片段”双模式返回

详情、编辑、新增相关页面继续保留现有 URL，但在 `X-Requested-With: XMLHttpRequest` 条件下只返回抽屉片段，在普通访问时仍可返回整页模板。

选择原因：
- 不破坏现有路由和直接访问能力
- 能兼容抽屉打开、整页刷新和错误回退三种场景

替代方案：
- 为抽屉单独新增 `/fragment` 路由
  - 放弃原因：会引入双路由维护成本，也容易让表单提交和跳转行为变复杂

### 3. 抽屉行为由通用脚本管理

新增统一抽屉脚本负责：
- 通过 Ajax 拉取抽屉内容
- 在右侧滑出详情/表单面板
- 提交抽屉内表单
- 成功后关闭抽屉并刷新左侧列表
- 按需初始化政策富文本编辑器

选择原因：
- 避免每个模块各写一套抽屉逻辑
- 可与现有列表局部刷新脚本解耦协作

替代方案：
- 将抽屉逻辑内嵌进每个页面模板
  - 放弃原因：维护成本高，后续一旦交互调整需要重复修改多个页面

### 4. 删除确认统一走自定义模态框

所有删除相关表单与批量删除按钮统一通过共享模态框脚本确认，不再使用浏览器原生 `confirm/alert`。

选择原因：
- 保持后台视觉一致性
- 同时覆盖单条删除和批量删除两个场景

替代方案：
- 每个页面单独写一个删除模态框
  - 放弃原因：重复模板多，后续文案或样式调整代价更高

### 5. 侧边栏菜单通过渲染层兼容旧数据

侧边栏将企业信息与招聘信息合并展示为一个父入口，并将政策管理显示为就业创业政策。优先在渲染层统一菜单数据，同时同步更新初始化 SQL，避免重新建库后名称回退。

选择原因：
- 当前本地数据库可能已经存在旧菜单数据，渲染层兼容可以立即生效
- 初始化数据同步修改可保证新环境一致性

替代方案：
- 只修改数据库初始化 SQL
  - 放弃原因：对已经存在数据的环境不会立即生效

## Risks / Trade-offs

- [抽屉与列表局部刷新脚本耦合增加] → 通过职责拆分处理：列表脚本只管列表刷新，抽屉脚本只管抽屉加载与抽屉内表单提交
- [Ajax 片段渲染与整页渲染可能出现模板分叉] → 保持同一模板双模式复用，片段仅通过 `th:fragment` 提取核心内容
- [政策富文本编辑器在抽屉中重复初始化] → 在脚本中增加幂等判断，只在抽屉内容首次加载后初始化
- [菜单改名后旧环境高亮规则失效] → 在侧边栏模板中同时支持父级路径与子级路径命中高亮
- [小屏下抽屉挤压列表] → 在响应式样式中将抽屉布局回退为上下堆叠
