## Context

当前后台已经具备首页、青年信息、企业信息、招聘信息、政策管理和分析页面的完整基础功能，但模板和样式仍以“通用管理后台”为主，尚未形成与目标设计稿一致的深色驾驶舱视觉体系。现有前端主要由 Thymeleaf 模板、单一的 `app.css` 和页面内少量 ECharts 配置组成，适合通过页面和样式层重构来完成一次跨模块视觉统一。

本次变更的核心约束已经明确：
- 优先修改页面，不调整后台接口、路由、服务和数据库结构。
- 继续使用现有 Thymeleaf 模型数据和表单提交方式，不引入新的前后端通信协议。
- 视觉目标以用户提供的设计稿为参考，重点拉齐整体壳子、列表页、分析页和表单页的呈现效果。
- 首页地图类区域如果缺少真实聚合数据，可先实现版式和视觉容器，不要求为此增加新接口。

## Goals / Non-Goals

**Goals:**
- 建立统一的后台驾驶舱风格视觉语言，覆盖导航、标题区、卡片、表格、表单、按钮和提示组件。
- 让首页、青年信息、分析页、企业信息、招聘信息、政策管理页面的布局更接近目标设计稿。
- 在不改变现有路由的前提下，将新增、编辑、详情页改造成更接近右侧抽屉面板的视觉结构。
- 复用现有分析数据，重写 ECharts 图表配置和页面编排，使分析页更接近目标稿的图表组合样式。
- 将页面改造控制在模板、CSS 和必要的页面内 JavaScript 范围内，保证业务功能不回退。

**Non-Goals:**
- 不新增或修改后台接口，不调整 controller/service/dao 的业务逻辑。
- 不为首页地图板块新增区域聚合数据接口或复杂地理统计能力。
- 不在本次变更中引入新的前端框架或拆分独立前端工程。
- 不追求设计稿逐像素还原，优先实现整体风格统一和关键页面相似度提升。

## Decisions

### 1. 采用“全局壳子 + 页面局部重排”的改造策略
先统一侧边导航、顶部标题区、面板、按钮、筛选栏和表格等全局骨架，再针对首页、列表页、分析页和表单页分别做局部布局重构。

选择原因：
- 当前所有模板都依赖公共 fragment 和同一份样式文件，适合先统一外层视觉语言。
- 可以快速让所有页面整体风格接近目标稿，再逐页提升细节还原度。
- 相比逐页零散修样式，更利于控制视觉一致性和后续维护。

备选方案：
- 逐个页面独立重写：不采用，容易出现风格割裂且重复样式较多。
- 新增整套主题文件并保留旧样式：暂不采用，当前页面数量可控，直接整理现有样式成本更低。

### 2. 保持现有路由与表单提交流程，表单页只做“抽屉化视觉”
新增、编辑、详情页面仍然保留现有访问路径和提交方式，但通过页面布局让表单看起来更像从右侧展开的操作面板。

选择原因：
- 能满足“不动后台接口”的要求。
- 避免为了模仿弹层/抽屉而引入额外异步加载接口。
- 风险低，现有保存、校验、回跳逻辑均可复用。

备选方案：
- 真正实现前端抽屉异步加载：暂不采用，虽然可行，但会引入更多页面内脚本和状态管理复杂度。

### 3. 分析页优先复用现有统计数据，前端重构图表类型与排版
分析页继续使用现有六类聚合数据，但前端将按目标稿重构为面积图、横向柱图、环图、饼图、堆叠图等混合布局。

选择原因：
- 当前后端已经能提供年龄、性别、学历、民族、政治面貌、创业需求等数据，足以支撑视觉重排。
- 不需要改接口即可显著提升页面接近度。
- 图表重构主要发生在页面层，回归验证成本可控。

备选方案：
- 为每张图单独新增接口和专门 DTO：不采用，与“优先只改页面”目标冲突。

### 4. 首页地图区采用“视觉优先、数据兼容”的容器方案
首页驾驶舱中的地图位优先实现设计风格一致的地图卡片或占位可视化容器；如果现有数据不足以支撑真实地理分布，则允许先展示静态底图、统一配色和标题结构。

选择原因：
- 当前首页接口未提供按地区分布的聚合结果，直接做真实地图会迫使后端扩展。
- 用户当前明确要求优先改页面，不优先动接口。
- 先把首页版式建立起来，后续再接入更真实数据时改动面更小。

备选方案：
- 暂时完全放弃地图位：不采用，会使首页与目标稿差异过大。

### 5. 通过统一样式变量和模块化类名控制页面一致性
继续使用单一 `app.css`，但要按主题变量、布局层、组件层、业务页层来重组样式命名和样式区域。

选择原因：
- 当前工程只有一份主样式文件，重组成本低。
- 保持资源结构简单，适合 Thymeleaf 项目。
- 能减少后续页面继续扩展时的样式冲突。

备选方案：
- 拆成多份 CSS 文件：可行，但本次不是必要条件，先在现有文件内完成结构化整理更稳妥。

## Risks / Trade-offs

- [跨多个模板和公共样式改动范围大] → 先统一全局骨架，再逐个模块调整，避免一次性大改导致页面全面回退。
- [首页地图缺少真实区域聚合数据] → 先实现与目标稿接近的视觉容器，不把真实地图数据作为本次交付前提。
- [表单页仍是独立页面，不是真抽屉] → 通过右侧面板式布局和弱化背景增强抽屉感，优先满足视觉目标。
- [单一 CSS 文件持续膨胀] → 在文件内按主题变量、公共组件、页面模块分段组织，减少后续维护成本。
- [不同模块表格字段差异较大] → 保持统一表格骨架和交互风格，仅在列编排上按模块做差异化展示。

## Migration Plan

1. 先重构公共页面壳子：侧边栏、顶栏、主题变量、通用卡片、筛选栏、按钮、表格和提示样式。
2. 再调整青年信息、企业信息、招聘信息、政策管理四类核心列表页，使整体视觉接近目标稿。
3. 调整青年、企业、招聘、政策的表单和详情页，使其呈现为右侧抽屉式布局。
4. 重构分析页图表布局和 ECharts option，验证现有数据在新样式下的可读性。
5. 最后调整首页驾驶舱版式，接入可复用的数据卡片、列表摘要和地图视觉容器。

回滚策略：
- 页面改造不涉及数据库和接口变更，可按模板和 CSS 文件级别快速回滚。
- 如果单页改造出现问题，可保留全局壳子并仅回退具体业务模板。

## Open Questions

- 首页地图区域最终是使用静态装饰底图，还是使用现有可获得的数据生成弱关联热度展示。
- 新增/编辑页是否需要在后续阶段进一步演进为真正的前端抽屉，而不仅是抽屉化页面布局。
