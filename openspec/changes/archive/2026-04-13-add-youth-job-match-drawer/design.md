## Context

当前系统已经支持从招聘岗位详情抽屉进入“匹配学生”结果抽屉，匹配逻辑直接复用 `YouthSearchRequest` 中已有的专业、学历、学校类别、学校标签四类条件。反向“学生匹配招聘”缺少对应入口，也缺少岗位查询侧对这四类条件的完整支持；现有 `JobSearchRequest` 和 `JobPostDao.xml` 仅覆盖岗位名称、企业、学历、经验、薪资、工作地区等常规筛选。

在交互上，学生详情当前通过右侧抽屉打开，抽屉关闭默认回到列表页。新需求要求在学生详情中继续打开匹配结果抽屉时覆盖原详情抽屉，并在关闭匹配结果后恢复到学生详情抽屉，避免管理员在学生与岗位之间切换时丢失上下文。

## Goals / Non-Goals

**Goals:**
- 在在校学生详情抽屉中提供“匹配招聘信息”入口。
- 让学生匹配招聘与岗位匹配学生使用完全一致的四类匹配条件: 专业、学历、学校类别、学校标签。
- 复用现有抽屉分页与局部刷新模式，新增岗位匹配结果抽屉。
- 支持抽屉覆盖链路: 学生详情 -> 匹配结果 -> 关闭后返回学生详情。

**Non-Goals:**
- 不新增“具体学校精确匹配”维度，不为岗位增加学校白名单关系表。
- 不引入权重打分、推荐排序或模糊匹配，仍采用已配置条件间 AND、同维度多值 OR 的显式过滤规则。
- 不改造现有岗位匹配学生的业务规则，只在其基础上补足反向能力与对称交互。

## Decisions

### 1. 反向匹配复用岗位搜索链路，而不是新增独立匹配服务
- 决定: 扩展 `JobSearchRequest` 与 `JobPostDao.xml`，让岗位搜索支持 `majorCodes`、`educationRequirements`、`schoolCategoryIds`、`schoolTagIds` 四类条件，再由学生匹配入口构造岗位查询条件。
- 原因: 这与现有“岗位 -> 学生”复用学生搜索链路的模式保持一致，后续若岗位列表页也要开放这些筛选条件，可直接复用同一查询能力。
- 备选方案:
  - 新增独立 `matchJobsForYouth()` DAO 查询: 语义更直接，但会复制分页、筛选与结果映射逻辑，长期维护成本更高。

### 2. 学生匹配条件由“青年记录 + 学校主数据”共同派生
- 决定: 学生详情匹配岗位时，从青年记录读取 `majorCode`、`educationCode`，再通过学校主数据读取 `categoryDictItemId` 与 `tagIds`。
- 原因: 青年记录本身并不持久化学校标签集合与学校类别 ID，但岗位匹配规则要求与岗位端条件完全一致；学校主数据已经是现成可信来源。
- 备选方案:
  - 在青年记录中冗余学校类别与标签: 会造成学校主数据变更后的同步问题，不适合当前模型。

### 3. 抽屉返回采用“来源 URL”链路，而不是直接关闭到列表
- 决定: 学生详情打开匹配结果抽屉时，为匹配结果请求携带来源详情地址；关闭匹配结果时优先恢复来源抽屉内容，而不是触发通用关闭逻辑直接回列表。
- 原因: 需求明确要求保留学生详情上下文，且现有系统已经大量使用抽屉加载片段，新增来源链路比引入全局抽屉栈状态更轻量。
- 备选方案:
  - 在前端维护完整抽屉栈: 灵活但复杂，会扩大当前抽屉脚本的职责。
  - 关闭匹配结果后重新打开学生详情整页: 会破坏抽屉局部交互体验。

### 4. 匹配结果页沿用现有结果抽屉模式
- 决定: 新增与 `job/match-results.html` 对称的学生匹配岗位结果模板，包含匹配条件摘要、总数、分页和岗位列表。
- 原因: 现有模板结构已验证可用，继续复用可以降低实现与测试成本，并让双向匹配在 UI 上保持一致。

## Risks / Trade-offs

- [岗位查询条件变复杂] -> 通过复用现有关联表 `job_post_education_rel`、`job_post_major_rel`、`job_post_school_category_rel`、`job_post_school_tag_rel` 的 `exists/in` 条件，避免引入新表和复杂数据迁移。
- [学生学校主数据缺失导致匹配条件不完整] -> 在设计上允许未解析出的维度按“未配置条件”处理，并在结果页展示实际生效的匹配摘要，避免误导管理员。
- [抽屉覆盖返回逻辑与现有关闭逻辑冲突] -> 将“来源详情抽屉恢复”限制在学生匹配岗位这一条链路中，尽量不改变其他抽屉关闭语义。
- [双向匹配规则理解不一致] -> 在结果页明确展示四类生效条件，确保管理员知道哪些维度参与了本次匹配。
