## Context

当前后台首页 `dashboard/index.html` 仅展示汇总卡片、业务摘要列表和按青年分类循环渲染的占位分布卡片，尚未真正提供空间分布地图。与此同时，青年信息表已经具备 `school_province_code`、`school_city_code`、`school_county_code` 以及 `native_province_code`、`native_city_code`、`native_county_code` 等三级区域编码字段，足以支撑按学校所在地和籍贯做聚合统计。

本次变更会同时涉及首页控制器、服务、MyBatis 聚合 SQL、首页模板、静态地图资源与区域主数据种子。项目中 `analytics/index.html` 已经接入 `ECharts`，因此首页地图无需重新引入新的图表框架，但需要解决两类数据和资源约束：

- 中国地图展示依赖全国省级区域名称与中国省级 GeoJSON 名称一致。
- 海东地图展示依赖海东市 2 区 4 县的区县主数据完整存在，并且与海东 GeoJSON 名称一致。

## Goals / Non-Goals

**Goals:**
- 在后台首页新增中国地图，展示在校大学生按学校所在省的分布情况
- 在后台首页新增海东地图，展示在校大学生按籍贯区县的分布情况
- 仅使用业务表实时聚合结果生成地图数据，不引入手工维护的静态统计表
- 在页面标题、图例或说明文案中明确两张地图的统计口径
- 让地图展示与当前区域主数据、青年信息编码结构保持一致，避免产生新的地区口径

**Non-Goals:**
- 不在本次变更中新增地图点击钻取、联动筛选或下钻详情页
- 不新增新的分析页面或独立 API 接口
- 不改变青年信息录入结构，也不新增新的区域字段
- 不在本次变更中处理在校大学生以外青年类型的地图展示
- 不在本次变更中引入第三方后端地图服务

## Decisions

### 1. 复用现有 `dashboard` 模块输出首页地图数据
首页地图直接在 `DashboardController` / `DashboardService` / `DashboardDao` 中补充聚合查询和模型注入，而不是新建单独分析模块或异步接口。

这样做的原因：
- 首页现有数据已经由 `dashboard` 模块统一装配，继续在同一链路补齐最简单
- 地图数据体量较小，和首页其他卡片一并渲染即可
- 避免新增额外路由、接口鉴权和前端二次请求逻辑

备选方案：
- 新增独立 `/api/dashboard/maps` 接口。放弃原因：当前需求规模较小，增加接口拆分会放大改造面

### 2. 中国地图和海东地图分别采用各自独立的统计口径
中国地图只统计 `youth_type = 'college'` 且按 `school_province_code` 聚合；海东地图只统计 `youth_type = 'college'` 且按 `native_county_code` 聚合，并通过 `sys_region` 关系限制在海东市下辖区县。

这样做的原因：
- 这是用户已经明确确认的业务口径
- “学校所在省”与“籍贯区县”表达的是不同分析视角，应各自独立实现和标注
- 继续沿用现有三级区域代码，不需要再增加衍生字段

备选方案：
- 两张图统一按同一字段聚合。放弃原因：会损失用户明确希望看到的两个不同业务视角

### 3. 首页地图继续使用 `ECharts`，地图边界以静态 GeoJSON 方式提供
前端继续复用项目已使用的 `ECharts`，通过静态资源目录提供中国省级 GeoJSON 与海东市区县 GeoJSON，页面加载后注册地图并渲染。

这样做的原因：
- 现有项目已具备 `ECharts` 使用经验和视觉风格，延续成本最低
- 静态 GeoJSON 部署简单，不依赖外部地图服务
- GeoJSON 与页面脚本一同版本管理，便于后续统一维护

备选方案：
- 接入在线地图服务。放弃原因：会引入外部依赖和网络不稳定因素，不适合当前后台系统

### 4. 区域名称匹配采用“主数据名称与 GeoJSON 名称对齐”的方式处理
聚合查询仍以 `sys_region.region_name` 作为地图数据项名称来源，但在落地前必须确保中国省级和海东区县名称与 GeoJSON 中的行政区名称一致；如存在口径差异，应优先调整种子数据或在地图层做最小化名称映射。

这样做的原因：
- `ECharts` 地图渲染以名称匹配最直接，当前系统也已有区域主数据表可复用
- 让主数据口径与前端边界资源统一，有利于后续继续做区域统计

备选方案：
- 在前端硬编码一套大量名称转换表。放弃原因：维护成本高，且容易与主数据继续漂移

### 5. 首页地图区替换当前占位卡片，但保留现有首页总体布局
当前首页的 `map-grid` 区域不再按青年分类循环输出占位卡片，而是改为两个固定地图卡片；其他统计卡片和列表摘要区保持不变。

这样做的原因：
- 用户需求只针对首页地图区域，不需要重做整页结构
- 保持现有首页信息密度和操作习惯，降低视觉变更风险

备选方案：
- 将地图下沉到分析页或重构整页为全屏驾驶舱。放弃原因：超出本次需求范围

## Risks / Trade-offs

- [区域主数据不完整] → 先补齐全国省级和海东市 2 区 4 县种子数据，并在实现前校验名称是否可匹配地图资源
- [GeoJSON 名称与数据库名称不一致] → 优先通过主数据或资源校对解决，必要时补充最小映射层
- [首页脚本体积增加] → 只在首页引入地图脚本和必要资源，避免影响其他页面
- [地图数据口径容易被误读] → 在卡片标题、副标题或图例中明确写出“按学校所在省”“按籍贯区县”
- [地图无数据时页面表现不清晰] → 需要在图表初始化时支持空数据兜底文案，避免空白区域造成误解

## Migration Plan

1. 补齐首页地图依赖的区域种子数据与静态 GeoJSON 资源。
2. 在 `dashboard` 模块增加两类聚合查询和模型装配逻辑。
3. 调整首页模板，用两个固定地图卡片替换当前占位分布卡片。
4. 完成首页地图渲染和空数据兜底处理。
5. 通过首页手工验证中国地图与海东地图均可正常渲染。

回滚策略：
- 若地图展示存在问题，可先回退首页模板与 `dashboard` 聚合逻辑，恢复当前占位卡片展示
- 新增的区域主数据和静态资源可保留，不影响旧首页逻辑恢复

## Open Questions

- 海东市地图 GeoJSON 的来源与精度需要在实施时确认；若暂无现成资源，需要先准备可商用或可内部使用的边界文件
