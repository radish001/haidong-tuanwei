## Context

当前 `sys_school` 和 `sys_major_catalog` 仅以名称作为唯一标识，`youth_info` 也直接保存 `school_name` 与 `major`；同时学历仍作为普通字典值维护，学位尚未进入青年受控字段体系。这导致学校或专业名称调整时，需要通过主数据管理逻辑批量回写青年信息；Excel 导入导出依赖名称下拉和名称匹配；分析统计通过字符串名称关联学校主数据，也无法统一承接教育部对学历、学位和专业体系的编码口径。

这次变更同时影响主数据管理、青年信息 CRUD、Excel 导入导出和在校大学生分析，是一次跨模块的数据标识体系升级。系统还需要兼顾两个现实约束：一是教育部官方编码是长期业务标识，不能退化成仅在主数据表内使用；二是当前后台管理员仍需要在页面和 Excel 中以“看得懂的名称”为主进行录入与核对。

## Goals / Non-Goals

**Goals:**
- 为学校、专业、专业分类、学历和学位引入稳定的编码化主数据口径，并要求后台管理同时维护编码和名称。
- 将青年信息中学校、专业、学历和学位的落库标识改为官方编码，名称只作为展示信息从主数据回显。
- 消除学校和专业名称变更时对青年业务数据的级联回写依赖。
- 统一 CRUD、Excel 导入导出和分析查询的关联口径，全部改为基于编码关联主数据。
- 为后续招聘岗位专业要求、多系统交换和标准数据对接提供稳定标识。

**Non-Goals:**
- 本次不引入外部同步接口，不负责自动从教育部系统抓取学校、专业、学历或学位目录。
- 本次不扩展招聘岗位专业要求、多选匹配逻辑，只为其提供基础编码体系。
- 本次不重构所有青年业务字段为外键 ID 体系，区域字段仍保持现有代码口径。
- 本次不新增新的分析图表，只调整现有分析聚合的数据关联方式。

## Decisions

### 1. 青年信息存官方编码，不存主数据表内部 ID

青年信息中的学校和专业将分别存 `school_code`、`major_code`，不存 `sys_school.id` 或 `sys_major_catalog.id`。主数据表仍保留 `id` 作为内部主键，但业务关联、Excel 口径和分析查询都以官方编码为准。

- Why:
  - 官方编码是跨环境、跨系统和跨时间更稳定的业务标识。
  - 避免不同环境下内部 `id` 不一致，影响导入导出和数据交换。
  - 和用户提出的“以教育部官方编码为准”目标一致。
- Alternatives considered:
  - 存主数据表 `id`：数据库 join 更直接，但业务语义弱，迁移和对外交换成本高。
  - 同时存 `id` 和编码：实现更复杂，容易出现双标识不一致，第一阶段收益有限。

### 2. 主数据表新增唯一编码字段，名称保留为展示字段

`sys_school` 新增 `school_code`，`sys_major_catalog` 新增 `major_code`，并建立唯一约束。专业分类和学历将从普通字典升级为“编码 + 名称”口径，学位新增同样的编码化主数据。名称字段继续保留，用于表单展示、筛选下拉和导出可读值。

- Why:
  - 管理员仍需要按中文名称进行维护和识别。
  - 保留名称字段可兼容现有后台展示方式，降低交互改动成本。
- Alternatives considered:
  - 只保留编码：会明显降低后台可操作性，也会使 Excel 使用成本上升。

### 3. 青年 CRUD 内部以编码写库，但页面仍展示名称

青年表单和列表不直接暴露“只看编码”的体验，而是通过受控主数据选项选择学校、专业、学历和学位。提交时传递编码，展示时通过编码关联学校名称、专业名称、学历名称和学位名称。

- Why:
  - 管理员更熟悉名称，不适合在主页面直接面对大量编码。
  - 可以在不牺牲用户体验的情况下完成底层数据标准化。
- Alternatives considered:
  - 页面完全改为编码录入：不利于使用，容易录错。
  - 页面继续传名称、后端再查编码：会保留名称驱动入口，弱化这次升级价值。

### 4. Excel 采用“可见名称、内部按编码校验”的导入导出策略

Excel 模板中的学校、专业、学历和学位仍以人类可读的选项形式呈现，但导入逻辑必须将选中的值解析到主数据编码，并在 `youth_info` 中保存编码。导出文件默认输出名称用于业务查看，必要时可在后续版本追加编码列，但本次保持模板核心可用性优先。

- Why:
  - 管理员批量处理 Excel 时更需要可辨认的名称。
  - 直接让管理员在 Excel 中维护官方编码，输入和核对成本过高。
- Alternatives considered:
  - Excel 只使用编码列：标准化最强，但业务可用性明显下降。
  - Excel 同时导出编码和名称双列：更完整，但会破坏当前模板结构并扩大变更范围。

### 5. 分析与统计改为通过编码关联主数据

所有依赖学校、专业和学历口径的聚合查询都应改为基于 `school_code`、`major_code` 和学历编码关联主数据；专业类别统计不再依赖青年信息中的冗余 `major_category` 字段，而是通过 `major_code` 关联专业主数据后反查专业分类。

- Why:
  - 现有学校分析和标签专题分析使用名称 join，最容易受到名称脏数据影响。
  - 当前 `major_category` 已存在 label/value 混用风险，直接取消青年侧冗余分类存储可以从根源避免不一致。
- Alternatives considered:
  - 只替换学校关联为编码，专业保持原状：会留下半套标准，后续匹配和分析仍要再次改造。

### 6. 采用“增量字段 + 数据回填 + 代码切换”的迁移方式

迁移策略以兼容为主：先为主数据和青年表增加编码字段并回填，再切换业务读写和分析查询，最后视实现情况决定是否保留原名称字段作为冗余展示字段或迁移辅助字段。

- Why:
  - 当前库中已存在大量名称数据，需要平滑迁移而不是一次性破坏式替换。
  - 允许在实现阶段分步骤验证主数据回填、青年 CRUD 和分析查询。
- Alternatives considered:
  - 直接删除名称字段并立即切换：风险高，回滚困难，且不利于定位迁移问题。

### 6. 专业分类不在青年信息中单独冗余存储

青年信息只保存 `major_code`，不再单独保存 `major_category` 或 `major_category_code`。任何列表筛选、导出展示和分析分组都通过 `major_code` 关联专业主数据，再反查专业分类主数据。

- Why:
  - 专业编码已经能唯一确定专业及其所属分类，单独保存分类会带来冗余和一致性维护成本。
  - 可避免“专业已变更但青年表中的分类未同步”的问题。
- Alternatives considered:
  - 继续保留 `major_category`：查询更直接，但会重复存储，仍需处理同步与回填风险。
  - 改存 `major_category_code`：比存名称更规范，但仍是冗余字段，价值低于直接由 `major_code` 反查。

## Risks / Trade-offs

- [历史数据中的学校/专业名称无法唯一映射到官方编码] → 先执行数据盘点与回填校验，对未匹配记录给出人工修复清单。
- [Excel 仍显示名称，可能出现重名学校或专业造成歧义] → 要求主数据层面编码唯一且名称冲突可见；如存在重名风险，可在模板选项中追加“编码 + 名称”组合展示。
- [代码切换期间同时存在名称字段和编码字段，可能出现双写不一致] → 在业务切换后只允许编码作为写入口，名称仅通过主数据衍生或迁移脚本维护。
- [分析 SQL 变更范围大，容易遗漏名称 join] → 对学校类别分析、学校标签分析、导入导出和青年列表查询分别建立针对性测试和核对清单。
- [专业类别筛选和统计 SQL 将比直接查青年表字段更复杂] → 统一通过 `major_code` 关联专业主数据封装查询，避免在多处复制复杂 join。

## Migration Plan

1. 为 `sys_school`、`sys_major_catalog` 增加编码字段及唯一约束，并补齐初始化数据或导入后的回填能力。
2. 将专业分类、学历从普通字典值口径升级为编码化主数据，并新增学位主数据。
3. 为 `youth_info` 增加 `school_code`、`major_code`、学历编码和学位编码字段，基于现有学校名称、专业名称、学历值、学位值回填历史数据。
4. 同步调整主数据管理页面和服务校验，要求学校、专业、专业分类、学历、学位新增/编辑必须提交编码。
5. 调整青年 CRUD、筛选、详情和列表展示逻辑，提交编码、展示名称。
6. 调整 Excel 模板、导入解析和导出逻辑，确保学校、专业、学历、学位按编码入库。
7. 调整列表筛选、导出和数据分析 SQL，使学校、专业和学历相关聚合全部改为按编码关联，并通过 `major_code` 反查专业分类。
8. 完成迁移核验后，评估是否保留原名称字段作为冗余展示或兼容字段；若保留，也必须明确编码为唯一写入来源。

Rollback strategy:
- 在字段切换前保留原名称字段与原逻辑，若新逻辑验证失败，可短期回退到名称驱动读取。
- 回填脚本和结构变更需可重复执行，便于重新初始化或重新导入主数据后再验证。

## Open Questions

- Excel 模板中的学校、专业、学历下拉，是否要显示为“编码 + 名称”，还是继续仅显示名称。
- 青年信息导出是否需要新增学校编码、专业编码、学历编码、学位编码列，还是继续只输出业务可读名称。
- 是否在迁移阶段直接移除 `major_category` 字段，还是短期保留为只读兼容字段后再清理。
