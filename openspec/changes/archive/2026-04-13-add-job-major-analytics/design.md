## Context

当前数据分析页由 `AnalyticsController` 和 `YouthAnalyticsService` 驱动，页面展示的图表数据全部围绕在校大学生画像展开。前端图表层已经具备统一的饼图渲染、Top N 聚合和点击跳转能力，但现有数据源中没有招聘岗位按专业聚合的统计结果。

另一方面，招聘岗位查询链路已经在后端支持 `majorCodes` 条件，但招聘列表筛选栏尚未暴露“专业要求”控件，因此即使分析页可以带着专业参数跳转，管理员也无法在列表页直观看到当前筛选状态。新需求要求在不重构整个分析模块的前提下，补充一个与“海东籍顶尖高校分析”并列的招聘专题区块，并确保从图表到招聘列表的交互闭环完整。

## Goals / Non-Goals

**Goals:**
- 在现有数据分析页中新增“招聘信息专业分析”专题标题和饼图区块。
- 基于招聘岗位专业关联数据输出专业分布，并仅保留前 15 个专业，其余合并为“其他”。
- 点击具体专业扇区后跳转到招聘信息列表页，并自动带入该专业筛选条件。
- 在招聘信息列表页显式展示并承接专业要求筛选，保证分析跳转后的状态可见、可继续修改。

**Non-Goals:**
- 不新增独立的“招聘分析”页签或整页分析模块。
- 不为“其他”聚合项提供多专业组合跳转能力。
- 不调整现有招聘岗位录入时的专业要求模型和关联表结构。
- 不扩展本次专题到招聘学历、地区或学校标签等其他招聘维度分析。

## Decisions

### 1. 在现有分析页中追加招聘专题区块，而不是新开分析页签
- 决定: 保持 `/analytics/college` 作为当前唯一可见分析入口，在页面下半部分新增与“海东籍顶尖高校分析”风格一致的专题标题和单个饼图区块。
- 原因: 这与用户要求一致，改动范围最小，也避免为了单张图引入新的页签切换和导航成本。
- 备选方案:
  - 单独增加招聘分析页签: 结构更清晰，但会扩大需求范围，并引出更多招聘分析入口设计问题。

### 2. 招聘专业统计按“岗位-专业关联”计数
- 决定: 基于 `job_post_major_rel` 关联 `job_post` 与 `sys_major_catalog` 聚合，每个专业统计命中该专业要求的有效岗位数量，`code` 使用 `major_code`。
- 原因: 这与点击后按专业筛选招聘岗位的行为完全对齐，数据口径清晰且复用现有表结构。
- 备选方案:
  - 直接从岗位主表按专业摘要字符串统计: 依赖展示字段，口径不稳定且无法可靠跳转到专业筛选。

### 3. Top 15 与“其他”聚合放在图表层处理
- 决定: 服务/DAO 返回完整专业分布，前端图表渲染时截取前 15 个专业并将剩余专业汇总为“其他”；“其他”不携带 `code`。
- 原因: 现有图表脚本已经有 Top N 聚合模式，复用前端聚合可以保持展示规则灵活，同时不影响后端统计结果的完整性。
- 备选方案:
  - 在 SQL 中直接裁剪前 15 名并汇总其他: 可以减少传输量，但 SQL 更复杂，也会把展示逻辑固化到查询层。

### 4. 招聘列表以多选专业筛选承接分析跳转
- 决定: 在招聘列表页筛选栏补充“专业要求”多选控件，并复用现有 `majorCodes` 查询参数；分析图表点击后跳转到 `/jobs?majorCodes=<majorCode>`。
- 原因: 后端查询已支持 `majorCodes`，补齐 UI 承接可以让分析跳转和手动查询共用同一条能力链路。
- 备选方案:
  - 仅接受隐藏 query 参数而不显示筛选控件: 实现更快，但用户无法确认当前过滤条件，体验不完整。

## Risks / Trade-offs

- [分析页同时展示学生与招聘专题，语义略混合] -> 通过独立标题区块和专题文案明确区分“招聘信息专业分析”。
- [专业数量较多导致饼图碎片化] -> 固定只展示前 15 名，并将剩余项合并为“其他”。
- [“其他”无法跳转到单一专业筛选] -> 明确设计为只展示不跳转，避免生成歧义筛选条件。
- [招聘列表新增多选筛选后布局更拥挤] -> 复用现有多选控件样式，优先保持与招聘表单一致的字段语义和展示方式。
