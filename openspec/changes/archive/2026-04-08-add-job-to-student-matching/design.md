## Context

当前招聘岗位只支持单值的学历要求，以及经验、薪资、工作地区等基础字段；岗位既不能表达多个专业或多个学历范围，也无法引用学校类别和学校标签。与此同时，在校大学生数据已经完成学校、专业、学历、学位编码化，学校类别和学校标签也存在于学校主数据中，这使“按岗位条件找学生”具备了数据基础，但缺少岗位侧条件建模和匹配入口。

这次变更横跨招聘、青年查询和主数据管理三个模块，既涉及多选条件存储，也涉及跨主数据的筛选 join 和删除约束，因此需要先明确数据模型与匹配语义，避免后续实现时出现字符串拼接、多值语义不一致或删除校验遗漏。

## Goals / Non-Goals

**Goals:**
- 为招聘岗位增加多选的专业要求、学历要求、学校类别、学校标签四类条件，并允许全部为空。
- 在招聘列表的顶部操作区提供“匹配学生”入口，基于选中的单条岗位查找匹配的在校大学生。
- 明确匹配语义：空条件忽略，不同维度按 AND 组合，同一维度按 OR 命中。
- 使专业要求匹配具体专业编码，学历要求按枚举命中，不做最低门槛推导。
- 补齐岗位条件引用主数据后的删除校验，避免岗位仍引用时误删专业、学校类别、学校标签或学历主数据。

**Non-Goals:**
- 本次不做岗位批量匹配学生，也不支持同时根据多条岗位合并匹配。
- 本次不引入智能推荐、权重评分、排序模型或模糊文本匹配。
- 本次不把 `jobCategory`、`jobDescription` 转为匹配条件来源；它们仍保留为普通岗位描述字段。
- 本次不扩展到毕业未就业青年、创业青年等其他青年分类，第一阶段仅面向在校大学生。

## Decisions

### 1. 岗位多选条件采用关系表存储

专业要求、学历要求、学校类别、学校标签都采用关系表存储，而不是在 `job_post` 上保存逗号分隔字符串或 JSON 文本。

- Why:
  - 多选条件后续要参与 SQL 筛选与删除校验，关系表更稳定、可索引、可约束。
  - 学校标签当前已经采用关系表模式，招聘岗位复用同类建模可以降低理解成本。
  - 避免字符串拆分、排序差异和脏数据导致的匹配失真。
- Alternatives considered:
  - 逗号拼接字符串：实现快，但查询、校验和迁移都容易失控。
  - JSON 数组：可读性尚可，但 MyBatis 与删除校验复杂度仍偏高。

### 2. 顶部操作区按钮基于“单条选中岗位”触发匹配

“匹配学生”放在招聘列表顶部操作区，与批量删除并列；管理员需要先勾选一条岗位，再执行匹配。未选中或选中多条时，系统提示先选择且仅选择一条岗位。

- Why:
  - 符合用户提出的“在操作区域加按钮”的交互预期。
  - 避免在每行操作列继续堆叠按钮，保持列表紧凑。
  - 匹配逻辑天然以单条岗位为输入，单选更容易解释结果。
- Alternatives considered:
  - 每行操作列增加按钮：更直接，但列表操作列会变得拥挤。
  - 支持多条岗位一起匹配：语义不清晰，结果解释成本高。

### 3. 匹配语义固定为“维度间 AND、维度内 OR、空值忽略”

岗位的四类条件都允许为空；只对已填写的条件参与筛选。若某维度选择了多个值，则学生命中其中任意一个即通过该维度。所有已填写维度都必须通过才算匹配成功。

- Why:
  - 这与后台多选筛选的常见认知一致，管理员容易预期结果。
  - 能保证岗位条件越完整，匹配结果越聚焦；条件缺失时又不会误伤结果。
- Alternatives considered:
  - 所有维度都必须命中全部选项：结果过窄，不适合学校标签等多选场景。
  - 任一维度命中即可：结果过宽，难以体现岗位要求。

### 4. 专业和学历匹配遵循显式值命中，不做推导

专业要求匹配具体专业编码，不通过专业类别或自由文本模糊匹配；学历要求按岗位显式选择的学历编码集合命中，不做“本科自动兼容硕士、博士”一类的层级推导。

- Why:
  - 用户已经明确要求专业匹配具体专业、学历走枚举命中。
  - 显式命中规则最容易在页面和结果中解释清楚。
- Alternatives considered:
  - 专业按专业类别匹配：会放大结果，降低岗位要求精度。
  - 学历按最低门槛推导：语义强依赖业务规则，且与多选设计相冲突。

### 5. 学校类别和学校标签通过学校主数据反查

匹配学生时不在青年表新增学校类别或学校标签冗余字段，而是通过 `youth_info.school_code -> sys_school -> 类别/标签关系` 完成筛选。学校标签默认采用“命中任一标签”语义。

- Why:
  - 学校类别和标签都已经存在主数据，不需要在青年表重复存储。
  - 保持学校标签主数据变更后，匹配行为自动跟随最新关系。
- Alternatives considered:
  - 在青年表冗余学校类别/标签：查询更简单，但一致性维护成本高。
  - 学校标签要求全部命中：过于严格，不利于多标签学校场景。

### 6. 匹配结果使用独立岗位匹配视图，但复用青年查询能力

从岗位列表进入匹配后，系统展示独立的“岗位匹配学生结果”视图，显示岗位摘要、已生效条件和学生列表；底层复用在校大学生查询能力扩展出来的筛选 join，而不是重复造一套学生查询逻辑。

- Why:
  - 独立视图更容易解释“这是由岗位条件筛出来的结果”，不会和普通青年检索混淆。
  - 复用青年查询能力能减少重复 SQL 和列表渲染逻辑。
- Alternatives considered:
  - 直接跳转普通青年列表并塞入查询参数：实现快，但岗位上下文容易丢失。
  - 完全独立实现学生查询：成本更高，后续易出现两套逻辑漂移。

## Risks / Trade-offs

- [岗位多选条件关系表增多，SQL 复杂度上升] → 通过统一的 DAO/Mapper 组装关系查询，避免在多个位置复制拼接逻辑。
- [学校标签采用“任一标签命中”可能比部分管理员预期更宽] → 在结果页明确展示已生效的匹配规则，并保留后续调整为更严格规则的空间。
- [顶部按钮依赖单选岗位，管理员可能误以为可以批量匹配] → 在按钮提示和校验消息中明确“请选择一条招聘信息进行匹配”。
- [主数据删除校验扩展后，更多删除操作会被拦截] → 在提示文案中明确指出是被招聘岗位引用，便于管理员理解和处理。

## Migration Plan

1. 为招聘岗位新增四类多选条件关系表，并补齐基础索引与初始化兼容脚本。
2. 扩展招聘岗位表单、详情回显和列表入口，使岗位可维护这些多选条件。
3. 扩展青年查询入参与 SQL join，支持基于专业、学历、学校类别、学校标签的岗位匹配筛选。
4. 增加岗位匹配结果视图与入口校验逻辑，仅允许基于单条选中岗位执行匹配。
5. 补充主数据删除校验，覆盖专业、学历、学校类别、学校标签被岗位引用时的拦截。

Rollback strategy:
- 关系表与匹配入口可独立回退，不影响既有招聘岗位基础维护。
- 如果匹配查询出现性能或语义问题，可先关闭“匹配学生”入口，保留岗位多选条件录入能力。

## Open Questions

- 匹配结果页是否需要再支持管理员手动追加二次筛选条件，例如地区或性别。
- 招聘列表是否需要直接展示四类新增条件摘要，还是仅在编辑页和匹配结果页展示。
