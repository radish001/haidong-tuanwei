## Context

当前青年信息使用统一的 `youth_info` 总表承载多类青年数据，前后端围绕 `YouthInfo`、`YouthFormRequest`、`YouthInfoDao.xml` 和青年信息表单/详情模板形成了一条稳定的维护链路。与此同时，青年信息 Excel 模板、导入和导出已经与页面维护保持同一字段顺序，因此新增业务字段不能只改数据库或表单，否则页面维护与批量导入导出会出现口径不一致。

“招考年份”是一个简单的业务属性，但它会跨越数据库结构、实体映射、MyBatis SQL、表单回填、详情展示、初始化数据和导入导出测试。该字段要求“仅年份”，因此需要在数据类型和校验方式上兼顾主库、H2 测试环境与 Excel 文本输入的兼容性。

## Goals / Non-Goals

**Goals:**
- 为青年信息新增可持久化的“招考年份”字段，并打通新增、编辑、详情、导入和导出链路。
- 使用对 MySQL 与 H2 都友好的存储方式，避免为了年份类型引入环境兼容问题。
- 保持现有青年列表和 Excel 结构的整体风格一致，让新增字段不会破坏已有流程。

**Non-Goals:**
- 本次不新增围绕招考年份的独立筛选、统计或分析面板。
- 本次不要求在所有青年列表页中强制展示招考年份列。
- 本次不引入新的字典、主数据或外部依赖。

## Decisions

### 1. 数据库存储采用整数年份列

采用 `recruitment_year` 整数列存储招考年份，而不是使用数据库原生 `YEAR` 类型。

原因：
- 当前项目同时维护 `schema.sql` 与 `schema-h2.sql`，整数类型在主库和 H2 中更容易保持一致。
- 年份只要求保存 4 位值，不需要完整日期语义。
- Excel 导入场景本身也更适合将年份作为文本/数字解析后再做业务校验。

备选方案：
- 使用 `YEAR`：数据库语义更直接，但测试环境与 ORM/Mapper 兼容性更差。
- 使用 `varchar(4)`：实现简单，但会降低数据约束强度，不利于后续排序、筛选和统计。

### 2. 字段先作为可空业务属性接入，不强制历史数据回填

新增字段在数据库和页面中默认允许为空，仅要求新录入或更新时可维护。

原因：
- 现有 `youth_info` 已有多类历史数据，无法保证都具备准确招考年份。
- 可空接入能降低上线和迁移风险，避免为了补字段阻塞当前需求。
- 后续若业务确认此字段成为必填项，可再通过单独变更收紧校验。

备选方案：
- 直接设为非空并回填默认值：会引入伪数据，影响后续业务可信度。
- 仅在页面层增加字段不落库：无法满足导入导出与统一数据管理要求。

### 3. 页面维护和 Excel 批量处理保持同一字段口径

招考年份同时接入青年信息表单、详情、Excel 模板、Excel 导入与 Excel 导出，保持单条维护和批量维护一致。

原因：
- 当前青年信息 Excel 模板与导出已经按统一字段顺序运作，新增字段若只进入页面，会造成人工录入和批量导入能力不对齐。
- 统一口径后，后续如果要增加筛选或统计，可直接复用同一数据源。

备选方案：
- 只改页面：短期实现少，但业务流程不完整。
- 只改 Excel：会导致抽屉表单与详情页看不到该字段。

### 4. 列表页先不强制新增显示列

招考年份先在新增/编辑/详情及导入导出中支持，不把它作为当前列表页必显列。

原因：
- 用户已明确反馈青年信息、招聘信息等表格过宽，当前列表的首要目标是收窄。
- 该字段更多是台账属性，详情页和导出文件更适合作为主展示位置。

备选方案：
- 同步加入列表：改动直观，但会进一步加宽列表。

## Risks / Trade-offs

- [年份字段输入不规范] → 在表单与导入逻辑中统一校验为 4 位年份，并在错误信息中明确提示。
- [历史环境升级遗漏新列] → 在 `schema.sql` 中同时补齐建表字段与增量 `alter table` 兼容语句。
- [Excel 列顺序变更影响现有模板校验] → 同步更新模板头、导入校验和测试用例，保持模板/导出/测试一致。
- [字段暂不进列表导致用户需要点详情查看] → 先满足数据录入与批量处理，若后续业务确认高频查看，再单独评估是否加入列表或筛选条件。
