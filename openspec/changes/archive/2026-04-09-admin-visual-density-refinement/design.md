## Context

当前后台已经完成 Bootstrap、Tom Select、TreeselectJS 的组件收敛，主要功能也能以统一的深色主题运行。但当前页面仍保留了较重的“壳层感”：

- 顶栏已经提供页面标题，列表页和分析页内部仍重复出现页面标题卡、说明卡和主卡头
- 列表页常见结构仍然是“标题卡 + 主卡 + 筛选区 + 操作区 + 表格”，层级偏多，纵向节奏偏散
- 首页和分析页存在多层卡片嵌套、较重阴影和较大圆角，更接近装饰性大屏而非正式政务驾驶舱
- 抽屉虽然已切到 Bootstrap `offcanvas`，但仍可继续收敛为更明确的“右侧工作台”体验
- `app.css` 已收缩到主题层，但当前 token 仍会带出较强的边框、阴影和留白感

这次变更不再处理 Bootstrap 接入本身，而是处理 Bootstrap 化之后的页面秩序问题。核心约束是：继续保留深色品牌基调，不改变业务流程，不更换已选插件，不引入新的前端框架，并确保桌面端和常规响应式场景都保持可用。

## Goals / Non-Goals

**Goals:**
- 将后台页面层级压缩为更简洁的“共享顶栏 + 主工作区”结构，减少重复标题和冗余说明块
- 统一列表页的页签、筛选、操作、表格、分页节奏，提升信息密度与对齐一致性
- 将首页和分析页整理为规则网格看板，弱化装饰性卡片，强化核心数据内容
- 将抽屉整理为统一的工作侧板体验，规范标题、正文、底部操作区的间距和对齐
- 调整全局主题 token，使整体视觉更接近正式、简洁、规整的政务驾驶舱风格

**Non-Goals:**
- 不新增业务功能、不修改接口或数据模型
- 不替换 Bootstrap、Tom Select、TreeselectJS、ECharts
- 不重做导航信息架构或菜单分组
- 不追求完全还原参考图的内容布局；参考范围仅限字体、格式、密度与版式秩序

## Decisions

### 1. 页面层级收敛为“共享壳层 + 单主工作区”

后台页面将优先保留共享 `topbar` 作为主标题表达层，页面内部仅在需要时保留轻量工作区标题，不再默认为每个页面额外增加大标题卡和重复说明块。

**Why:** 这样可以减少纵向占用和重复文案，让管理员进入页面后更快聚焦在可操作内容上。  
**Alternative considered:** 保留当前“顶栏 + 页面标题卡 + 业务卡”三层结构，仅缩小字号和间距。该方案会继续保留层级冗余，无法根本解决信息分散问题。

### 2. 列表页统一为紧凑工具台结构

列表页将统一为“页签或切换区 + 紧凑筛选工具栏 + 表格 + 分页”的工作台结构：

- 页签和工具栏尽量贴近
- 筛选项与操作按钮共享同一主卡区域
- 列表主卡只保留必要的卡头，不重复表达“XX列表”含义
- 表格、分页、每页条数切换统一对齐和密度

**Why:** 这是最接近政务后台日常使用模式的结构，也能最大化利用 Bootstrap 的 `nav`、`row`、`table`、`pagination`。  
**Alternative considered:** 保留各模块当前各自的筛选区和按钮区布局，再分别微调。该方案很容易继续造成模块间风格漂移。

### 3. 首页和分析页采用规则网格看板

首页与分析页将使用统一的栅格节奏组织地图、统计卡和图表，不再通过多层卡片嵌套制造“主块感”。图表卡以规则网格为主，说明文本最小化，主要焦点放在数字、图形和主要操作。

**Why:** 参考方向的核心不是更强装饰，而是更稳定的网格秩序和更高的内容聚焦度。  
**Alternative considered:** 继续通过渐变、阴影和大圆角强化视觉冲击。该方案会让页面更“花”，但不会更整齐。

### 4. 抽屉定位为工作侧板，而非第二页面

`offcanvas` 将继续作为统一抽屉容器，但视觉和结构上明确为“工作侧板”：

- 头部更薄，标题更直接
- 表单区使用更规整的 Bootstrap 栅格
- 底部操作区固定清晰，减少装饰性说明
- 桌面端保留足够宽度，移动端自然收缩为全宽抽屉

**Why:** 这符合管理员边看列表边处理数据的工作模式，也最符合 Bootstrap `offcanvas` 的预期用法。  
**Alternative considered:** 将抽屉重新做成更完整的详情页风格。该方案会再次引入页面层级重复。

### 5. `app.css` 仅调整视觉 token，不回退为结构样式文件

这次调整仍然约束 `app.css` 只处理主题 token 与少量页面级视觉覆盖，包括：

- 页面整体间距
- 卡片圆角和阴影强度
- 描边、表头、工具栏背景层级
- 标题字号与字重
- 表格密度与抽屉内间距

不会重新在 `app.css` 中引入一套独立于 Bootstrap 的通用布局结构。

**Why:** 视觉整理的目标是降噪，不是再长出第二层组件系统。  
**Alternative considered:** 通过新增更多项目类名来定制每个页面。该方案短期可控，长期会再次失去边界。

## Risks / Trade-offs

- [减少页面标题和说明块后，部分页面可能显得“信息不够明确”] → 优先保留真正承担工作语义的标题，只删除重复表达
- [压缩筛选区和工具栏后，小屏可能更容易换行] → 先按 Bootstrap 栅格约束断点，再为筛选项设置合理优先级
- [首页和分析页减弱装饰后，品牌感可能下降] → 保留深色主题、顶栏质感和少量品牌高亮，但弱化过重阴影和大面积装饰
- [抽屉进一步收紧后，复杂表单可能出现空间紧张] → 保留桌面端较宽的 `offcanvas` 宽度，并在表单内部使用更稳定的双栏栅格
- [不同模块的历史文案与区块数量差异较大，统一节奏时容易出现例外] → 先定义共享节奏，再允许少量页面级补充，不反向推动共享规则失控

## Migration Plan

1. 先整理共享层 token 和共享版式规则，例如顶栏高度、内容区边距、卡片半径、阴影和表格密度
2. 再按列表页统一骨架整理青年、招聘、企业、政策、系统字典等主要工作台页面
3. 随后整理首页和分析页的图表/地图网格，压缩重复标题和次要说明
4. 最后整理抽屉内表单与详情页的头部、正文、底部操作区节奏，并做主要断点检查
5. 如需回滚，可按模板、共享样式和 token 调整分层回退；本次不涉及后端数据和接口迁移

## Open Questions

- 首页是否仍需保留独立欢迎标题区，还是完全将“页面名”收敛到共享顶栏
- 分析页是否需要保留模块内二级标题说明，还是改为更轻量的图表标题体系
- 某些业务模块是否仍需要保留少量提示性副标题，以满足政务场景下的解释需求
