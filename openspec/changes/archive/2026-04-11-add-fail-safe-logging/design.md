## Context

当前系统基于 Spring Boot 默认日志能力运行，项目中没有独立的 `logback-spring.xml` 文件日志配置，关键业务模块也基本没有形成统一的业务日志规范。与此同时，`MyBatis` 仍通过 `StdOutImpl` 直接输出 SQL 到标准输出，导致 SQL 日志无法与应用日志统一控制、归档和分级。

这次变更是一个跨模块的横切改动，至少会触及认证、青年信息、招聘信息、系统设置、数据分析等多个 controller 和 service，以及全局日志配置。需求重点不仅是“能写日志”，还包括两个约束：

- 日志文件必须支持按月归档，并在单月文件达到阈值后自动拆分。
- 日志处理错误不能阻碍主业务系统，不能因为日志写入、滚动或归档异常破坏正常请求链路。

## Goals / Non-Goals

**Goals:**
- 为应用建立统一的文件日志方案，满足按月归档和按大小切分。
- 将应用日志、异常日志和 SQL 日志统一纳入 SLF4J / Logback 体系。
- 在关键业务操作中补充最小必要日志，支持问题定位、审计和导入导出排障。
- 明确并落实“日志故障隔离”原则，避免日志子系统异常升级为业务失败。

**Non-Goals:**
- 不引入 ELK、OpenSearch、Kafka、远程日志中心等外部日志基础设施。
- 不做全量 SQL 审计，也不默认打开高噪音 debug 级别日志。
- 不改变现有业务接口、数据库表结构、页面功能和权限模型。
- 不在本次变更中实现复杂的链路追踪、traceId 全链路治理或监控告警平台接入。

## Decisions

### 1. 使用 Logback 文件滚动作为统一日志方案

决定新增 `logback-spring.xml`，使用 Spring Boot 默认集成的 Logback 作为唯一日志实现，不额外引入新依赖。

选择原因：
- 项目已经基于 Spring Boot，Logback 为默认方案，落地成本最低。
- 支持 `SizeAndTimeBasedRollingPolicy`，天然满足“按时间周期 + 按大小切分”需求。
- 配置能力足够覆盖控制台、文件、级别隔离和包级日志控制。

备选方案：
- Log4j2：能力也满足，但需要额外替换默认日志实现，改造收益不高。
- 仅使用 `application.properties` 配置文件日志：难以表达复杂滚动策略和差异化 logger 控制。

### 2. 日志文件按月归档，并在单月内按大小分片

决定使用 `SizeAndTimeBasedRollingPolicy`，文件命名模式采用类似 `logs/app.%d{yyyy-MM}.%i.log`。

选择原因：
- `%d{yyyy-MM}` 能直接形成“每月一个归档周期”的运维视角。
- `%i` 支持在同一个月内按大小自动拆分多个文件。
- 策略足够清晰，便于后续扩展压缩、保留周期和总容量限制。

备选方案：
- 按天滚动：粒度更细，但不符合当前“每月一个文件周期”的要求。
- 仅按大小滚动：缺少月维度归档，不利于运维检索和归档管理。

### 3. 日志故障默认降级，不允许反向影响业务请求

决定在业务代码保持“旁路记录”原则的基础上，为文件日志增加一个 fail-safe rolling file appender。这样即使文件 appender 在启动期或运行期发生路径、权限或文件打开错误，系统也会降级为控制台日志 + 文件日志静默失效，而不是让 Spring Boot 将日志配置错误升级为应用启动失败。

落实方式：
- 增加一个轻量级的 fail-safe file appender 包装原有 rolling file appender 行为。
- 业务逻辑中不捕获日志异常后重新抛出业务失败。
- 文件日志异常时，允许日志系统丢失文件日志并退回控制台输出，但业务流程继续执行。
- 对导入、导出、增删改等关键路径，只记录摘要，不在日志中执行额外昂贵逻辑。

选择原因：
- 这满足“日志处理错误不能阻碍主业务系统”的核心要求。
- 避免把日志从可观测性能力变成业务可用性的单点依赖。

备选方案：
- 完全依赖原生 `RollingFileAppender`：在 Spring Boot 下文件 appender 初始化错误会被视为致命日志配置错误，无法满足“不阻断主业务”的要求。

### 4. MyBatis SQL 日志统一切换到 SLF4J

决定移除 `StdOutImpl` 直出方式，改为由 SLF4J 承接 SQL 日志，并在 Logback 中统一控制级别和去向。

选择原因：
- 避免 SQL 日志绕过文件滚动体系。
- 可以按包名或 mapper 范围精细控制是否输出 SQL。
- 便于在测试、开发、生产环境间切换噪音级别。

备选方案：
- 保持 `StdOutImpl`：实现最简单，但无法满足统一归档和统一治理。

### 5. 业务日志采用“关键动作摘要”策略

决定先在高价值路径补日志，而不是对所有方法机械加日志。

优先覆盖：
- 认证加载与登录失败场景
- 青年信息新增、更新、删除、批量删除、导入、导出
- 招聘信息新增、更新、删除、批量删除
- 系统设置中主数据维护和分析配置变更
- 全局异常处理入口

日志内容聚焦：
- 操作人、业务对象、对象标识、动作、结果、数量、失败原因摘要

不记录：
- 密码、原始 Excel 内容、完整敏感请求体、超大对象序列化内容

## Risks / Trade-offs

- [日志量增长] → 只记录关键业务摘要，避免对列表查询、循环明细逐条打 info 日志。
- [磁盘占用不可控] → 配置单文件大小阈值、保留历史上限和总容量上限。
- [SQL 日志噪音过高] → SQL 日志通过包级别控制，默认不开启高噪音明细。
- [日志丢失与业务可用性之间的取舍] → 优先保证业务可用性，接受日志子系统异常时的部分日志缺失。
- [埋点范围过大导致改动分散] → 第一阶段只覆盖认证、核心增删改、导入导出和异常处理。

## Migration Plan

1. 新增 `logback-spring.xml`，保留控制台输出并增加滚动文件输出。
2. 调整 `application.properties` 中的 MyBatis 日志接入方式，统一走 SLF4J。
3. 在关键 controller/service 增加最小必要日志，并在异常处理入口补充 warning/error。
4. 本地验证应用启动、日志落盘、月份命名和大小切分配置正确。
5. 若上线后出现日志配置问题，可临时回退到默认 Spring Boot 日志配置，业务功能本身不受影响。

## Open Questions

- 单文件大小阈值最终取值是 `50MB`、`100MB` 还是其他值。
- 日志保留周期按月保留多久更符合部署环境的磁盘约束。
- 是否需要在本次范围内拆分独立的 `error` 日志文件，还是先统一到主应用日志。
