## Why

当前项目只有默认控制台日志，缺少统一的文件归档、滚动切分和关键业务日志埋点，线上排查问题与审计操作都比较困难。随着系统逐步完善，需要建立可运维的日志体系，同时确保日志写入异常或归档异常不会阻断认证、增删改、导入导出等核心业务流程。

## What Changes

- 为系统引入统一的文件日志配置，支持按月归档并在单月日志超过阈值时自动切分文件。
- 将现有日志输出统一收敛到应用日志框架，替代直接输出到标准输出的 SQL 日志方式。
- 在认证、青年信息、招聘信息、系统配置、分析等关键业务入口补充最小必要的业务日志，覆盖成功、失败、异常和批量操作摘要。
- 增加日志故障隔离要求，确保日志写入、滚动或归档异常时主业务请求仍按既有业务逻辑继续处理，不因日志处理失败而中断。

## Capabilities

### New Capabilities
- `application-logging`: 定义系统文件日志归档、关键业务日志记录及日志故障不影响主业务的要求

### Modified Capabilities
- None

## Impact

- 受影响代码主要包括 `src/main/resources/application.properties`、新增日志配置文件，以及若干 controller/service 的业务日志埋点。
- 受影响的运行行为包括应用日志落盘方式、SQL 日志接入方式、关键操作的运行时可观测性。
- 不涉及对外 API 变更，不改变现有业务页面和数据模型。
